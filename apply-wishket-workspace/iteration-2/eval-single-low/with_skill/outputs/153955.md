## 지원서 본문

안녕하세요. React/TypeScript 7년차 풀스택 개발자입니다.

[프로젝트 분석]
본 프로젝트의 핵심은 다음과 같이 파악됩니다:
1. "아지트" 기반 개인화된 음성 채팅 공간 — 생성·초대·참여 흐름이 자연스러워야 사용자 유지율 확보
2. 실시간 음성 통신의 안정성 — 동시 접속자 수에 따른 미디어 서버 부하 관리가 서비스 품질의 핵심
3. MVP 범위 설정 — 기획·디자인·개발·3개 플랫폼·관리자 페이지 전체를 90일 안에 완성하려면, 핵심 기능과 후순위 기능을 명확히 나누는 것이 프로젝트 성패를 좌우

예상되는 주요 이슈:
- 음성 채팅 품질 관리: 네트워크 상태에 따른 끊김, 에코, 지연 등이 사용자 이탈의 직접 원인 — 미디어 서버 선정과 네트워크 예외 처리 설계가 초기부터 필요
- 동시 접속 확장성: 아지트당 참여 인원이 늘어날수록 미디어 스트림 관리 복잡도가 급증 — SFU 구조 기반으로 설계해야 향후 확장 가능
- 3개 플랫폼 동시 개발: PC 웹, Android, iOS를 별도로 개발하면 90일 내 불가 — 크로스 플랫폼 전략이 필수

[유사 프로젝트 경험]
1) 제철소 실시간 데이터 처리 시스템
- 공장 현장에서 초당 수백 건의 센서 데이터가 유입되는 실시간 대시보드 개발
- 실제로 겪은 이슈: 데이터 유입 속도가 화면 가공 속도를 초과하여 화면 멈춤 발생 → 전처리 필터링 + 배치 렌더링으로 DOM 업데이트 최소화, 중간 데이터를 메모리 파이프라인으로 전환하여 디스크 누적 문제 해결
- 결과: 유입 속도에 밀리지 않는 실시간 처리 성능 확보, 설정 파일 기반 규칙 엔진으로 코드 수정 없이 신규 대응 가능

2) HR 관리 SaaS — 멀티 플랫폼 + 권한 체계
- 근태·급여 관리 서비스를 웹·Android·iOS 3개 플랫폼으로 4년간 운영
- 실제로 겪은 이슈: iOS/Android 간 WebView 동작 차이(쿠키, 세션, 스크롤)로 통합이 어려운 상태 → 플랫폼별 분기를 추상화 레이어로 격리, 비즈니스 로직은 공유하고 네이티브 연동부만 분리
- 결과: 단일 코드베이스로 3개 플랫폼 서비스 운영, 5단계 권한 체계 기반 관리자 시스템 안정 운영

[진행 프로세스]
1단계. 기획·설계 — 2주
- 아지트 생성·초대·참여 흐름 정의, 화면 설계
- 음성 통신 아키텍처 설계 (SFU 기반 미디어 서버 선정)
- 기술 스택 확정 및 MVP 기능 우선순위 확정
- 산출물: 기능 명세서, 화면 설계서, 아키텍처 다이어그램
- 주 2회 진행 공유 (화면 설계 초안 포함)

2단계. 핵심 기능 개발 — 7주
- 아지트 생성·초대·참여 + 실시간 음성 채팅 구현
- 크로스 플랫폼 앱 개발 (React Native 기반 Android/iOS)
- 백엔드 API + 인증 + 관리자 페이지
- 주 1회 진행 공유 + 중간 데모 2회 (3주차, 6주차)

3단계. QA·수정·배포 — 4주
- 음성 품질 테스트 (다양한 네트워크 환경)
- 스토어 배포 준비 + 운영 환경 구축
- 버그 수정 및 사용성 개선
- 산출물: 배포 완료된 웹/앱 + 관리자 페이지 + 운영 문서

[미팅 시 논의 희망 사항]
- 아지트 동시 접속 최대 인원 규모 — 음성 채팅은 인원 수에 따라 아키텍처가 달라지므로 초기 확정 필요
- MVP에서 아이템 구매 등 비즈니스 모델 기능의 포함 여부 — 90일 내 핵심 기능 완성도를 높이려면 후순위 분리 권장
- 아지트의 상시 존재 여부 — 상시 존재 시 서버 리소스 관리 전략이 달라지므로 설계 단계에서 방향 확정이 중요

## 클라이언트 질문 답변

> 동시 접속 최대 인원 규모와 아지트의 상시 존재 여부

동시 접속 인원에 따라 음성 통신 아키텍처가 크게 달라집니다. 소규모(10명 이하)는 P2P mesh로도 가능하지만, 그 이상이면 SFU(미디어 서버) 기반이 필수입니다. 아지트 상시 존재 여부는 서버 리소스 설계에 직결되므로, 미팅에서 예상 규모와 함께 논의하면 최적의 구조를 제안드리겠습니다.
