[프로젝트 분석]
본 프로젝트의 핵심은 다음과 같이 파악됩니다:
1. WebRTC 기반 실시간 음성 채팅의 안정적 구현 — 다수 사용자 동시 접속 시 지연 없는 음성 품질 확보
2. "아지트" 중심의 커뮤니티 구조 설계 — 방 생성, 초대, 참여자 관리 등 사용자 간 관계 모델
3. MVP 범위 선정과 확장 가능한 구조 — 향후 아이템 구매 등 비즈니스 모델 확장을 고려한 기반 설계

예상되는 주요 이슈:
- 동시 접속 규모에 따른 미디어 서버 전략: P2P(소규모)와 SFU(다수) 간 전환 기준이 사전에 정의되지 않으면 이후 아키텍처 변경 비용이 큼
- 모바일 웹 환경의 음성 입출력 제어: iOS Safari/Android Chrome 간 마이크 권한, 백그라운드 오디오 정책이 상이하여 플랫폼별 대응 필요

[유사 프로젝트 경험]
1) 제철소 실시간 품질 분석 시스템 — 실시간 대량 데이터 처리
- 제철소 공정 데이터를 실시간으로 수집·분석하는 데스크탑 시스템 4년 운영
- 실제로 겪은 이슈: 실시간 데이터 유입 속도가 화면 가공 속도를 초과하여 처리 지연 누적 → 전처리 필터링과 배치 렌더링으로 DOM 업데이트 최소화, 중간 데이터를 메모리 파이프라인으로 전환하여 최종 결과만 저장
- 결과: 데이터 유입 속도에 밀리지 않는 실시간 처리 성능 확보, 신규 공정 대응 시 설정 파일만으로 처리 가능한 규칙 엔진 구축

2) HR 관리 SaaS — 멀티플랫폼 + 권한 체계
- 근태/급여 관리 SaaS의 프론트엔드를 4년간 설계·운영, React Native + WebView 하이브리드로 웹/Android/iOS 3개 플랫폼 서비스
- 실제로 겪은 이슈: 5단계 사용자 권한(어드민→사업주→관리자→근로자)이 화면 곳곳에 산재하여 신규 권한 추가 시 전체 페이지 검토 필요 → 선언적 권한 기반 렌더링으로 중앙 집중화, API 레벨 이중 검증 적용
- 결과: 이 아키텍처 위에 6개 모듈 안정적 신규 개발 완료, 단일 코드베이스로 3개 플랫폼 운영

[진행 프로세스]
1단계. 기획/설계 — 2주
- 요구사항 정의, 화면 설계, ERD, API 명세
- 동시 접속 규모 기준 미디어 서버 전략(P2P vs SFU) 확정
- 산출물: 화면설계서, ERD, API 문서
- 주 2회 진행 공유 (화/금)

2단계. 디자인 + 핵심 개발 — 6주
- UI/UX 디자인 병행, 음성 채팅 핵심 모듈 우선 개발
- 아지트 생성/참여, 실시간 음성 연결, 사용자 인증
- 중간 데모 2회 (3주차, 5주차)

3단계. 관리자 + QA + 배포 — 5주
- 관리자 페이지, 모바일 웹 크로스 브라우저 대응
- 통합 QA, 수정, 서버 인프라 구성 및 배포
- 최종 산출물: 기획서, 디자인 파일, 소스 코드 전체, 배포 환경

[미팅 시 논의 희망 사항]
- 아지트당 동시 접속 최대 인원 기준 — 미디어 서버 전략(P2P/SFU)과 인프라 비용에 직접 영향
- MVP 기능 우선순위 — 90일 내 기획+디자인+개발 전체를 한 명이 수행하므로 핵심 기능 우선순위 합의 필요
- 음성 채팅 중 네트워크 불안정 시 재연결 정책 — 사용자 경험에 큰 영향을 주는 부분으로, Discord 수준의 자동 재연결이 MVP 범위에 포함되는지 사전 합의 희망
