## 지원서 본문

안녕하세요. React/TypeScript 7년차 풀스택 개발자입니다.

[프로젝트 분석]
본 프로젝트의 핵심은 다음과 같이 파악됩니다:
1. 출제자-학생 양면 마켓플레이스 구조 — 콘텐츠 업로드/검수/결제/다운로드 흐름이 끊김 없이 연결되어야 함
2. PDF 콘텐츠 보호 — 결제 후 워터마크 삽입 다운로드, 미결제 상태에서의 접근 차단이 서비스 신뢰의 핵심
3. 확장 가능한 카테고리/메타데이터 설계 — 현재 국어·사회·과학이지만 향후 과목 추가에 유연하게 대응해야 하는 구조

예상되는 주요 이슈:
- PDF 워터마크 처리 성능: 결제 완료 시점에 실시간 워터마크 삽입은 사용자 대기 시간에 직결 — 비동기 처리 또는 사전 생성 전략이 필요
- 다중 조건 필터링 + 검색 성능: 과목·학년·키워드·가격 등 필터가 복합적으로 작동할 때, 데이터 증가에 따른 쿼리 성능 저하 대비 필요

[유사 프로젝트 경험]
1) B2B SaaS 플랫폼 — 역할별 접근 권한 체계 설계 및 6개 모듈 신규 개발
- 근태/급여 관리 SaaS에서 어드민→사업주→관리자→근로자 5단계 권한이 존재했으나, 권한 분기 로직이 곳곳에 산재하여 신규 기능 추가 시 전체 페이지 검토가 필요한 상태
- 실제로 겪은 이슈: 권한 경계가 불명확하여 데이터 접근 오류 발생 → 권한 로직을 중앙 집중화하고, 서버·프론트 양단 이중 검증 구조로 전환
- 결과: 이 구조 위에 6개 모듈을 안정적으로 신규 개발 완료, 신규 권한 추가 시 설정만 변경하여 영향 범위 최소화

2) 시뮬레이션 플랫폼 — 데이터 부재 상태에서 분석 체계 구축
- 124개국 사용자가 접속하는 SaaS 플랫폼에서 기능 우선순위, 다국어 대응 시기 등 제품 방향성이 감에 의존하는 상태
- 실제로 겪은 이슈: 사용자 행동 데이터가 전혀 없어 어떤 기능을 우선 개발할지 근거가 없음 → 분석 파이프라인을 처음부터 구축하여 클라이언트·서버 양쪽 이벤트 추적 체계 수립
- 결과: 사용자 이탈 단계를 정량적으로 파악, 국가별 유입 패턴 기반으로 타겟 시장 우선순위 도출 — 이 경험이 본 프로젝트의 콘텐츠별 구매 전환율 분석, 인기 카테고리 파악에 직접 활용 가능

[진행 프로세스]
1단계. 기획/설계 — 7일
- 요구사항 기반 기능 명세서, 화면 설계서(와이어프레임), ERD, API 명세서 작성
- DB 구조는 과목 추가에 유연하도록 카테고리 테이블 분리 설계
- 주 2회 진행 공유, 기획 완료 시 리뷰 미팅

2단계. 디자인 + 핵심 기능 개발 — 20일
- UI/UX 디자인 (학생 페이지, 출제자 페이지, 관리자 페이지)
- 인증/회원, 콘텐츠 CRUD, 검색/필터링, PDF 업로드/워터마크 처리 구현
- 중간 데모 1회 (디자인 시안 + 핵심 기능 동작 확인)

3단계. 결제 연동 + 관리자 + QA — 23일
- PG사 결제 연동 (장바구니, 바로구매), 정산 관리
- 관리자 대시보드, 콘텐츠 검토/승인, 회원/결제 관리
- 반응형 웹 대응, 보안 점검, QA 및 수정
- 주 2회 진행 공유, 최종 산출물 전달 및 배포

[미팅 시 논의 희망 사항]
- PDF 워터마크 방식: 다운로드 시점 실시간 생성 vs 결제 확정 후 비동기 사전 생성 — 사용자 경험과 서버 부하 간 트레이드오프 논의
- 출제자 정산 주기 및 방식 (월 정산, 건별 정산 등) — 정산 모듈 설계에 영향
- 향후 앱 확장 계획 여부 — 현재 반응형 웹 기준이지만, 앱 전환 가능성이 있다면 API 설계 시 고려 가능
