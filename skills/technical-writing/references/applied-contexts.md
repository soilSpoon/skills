# 실무 적용 — PR · 설계 문서 · README · 커밋 · 문서 세트

원칙은 정식 문서뿐 아니라 개발자의 일상 글쓰기에 그대로 적용된다("한 페이지짜리 간단한
문서에도 적용"). 각 맥락은 어떤 *유형*에 가까운지로 골격이 정해진다.

## PR 설명 (본질: "문제 해결" 문서)

토스 PR 템플릿 자체가 가이드의 화신이다 — **수정한 내용 / 변경이 필요한 이유 / 체크리스트.**

- **가치 먼저:** 첫 줄에 "무엇을·왜"(해결한 문제·효과)를 쓴다. diff 나열(기능 중심) 금지.
- **유형대로:** 문제 정의 → 해결 방법 → **확인 방법**(테스트/스크린샷).
- **구체성:** "성능 개선" 대신 "응답 1초 → 200ms".
- **능동·주체:** "버그가 수정됨" 대신 "X 시 Y가 발생하던 버그를 Z로 고침".
- **메타담화 제거:** "이번 PR에서는 ~하려고 했습니다" → 바로 본론.
- **과정 서술 금지:** 분리 이력·시행착오가 아니라 *리뷰어가 알아야 할 결과*만.

## 설계 문서 / ADR (본질: "설명" 문서)

- *등장 배경·해결할 문제*를 먼저(가치 먼저). 의사결정 과정·대안 비교를 명시(설명 원칙).
- 다이어그램을 적극 사용(ASCII 포함). 새 개념은 정의(자세히 설명).
- **한 문서 하나의 결정.** 제목은 결정 내용을 평서문 키워드로.

## README (본질: "시작하기" + 일부 "참조")

- **가치 먼저:** "이 라이브러리로 무엇을 할 수 있는가". 개요 필수.
- 설치 → 실행 → 확인의 **막힘없는 경로**, 실행 가능한 코드.
- 일관된 섹션(설치/사용법/API). H4까지 깊어지면 별도 `docs/`로 분리(한 페이지 하나 + 크로스링크).

## 커밋 메시지 (제목 = "효과적인 제목"의 축소판)

- 제목: 핵심 키워드 · 간결(~50자) · 평서문 · 형식 통일(`feat:`/`fix:` 등 동사형).
- 본문: "왜"(가치·배경)를 능동·구체로. 모호어·메타담화 금지.

## 문서 세트 구조 (여러 페이지)

1. 유형을 파악하고 → 2. 좌측 내비(디렉터리)로 구조 설계
   (`tutorials/ how-tos/ explanations/ reference/ troubleshooting.md glossary.md`) → 3. 크로스링크.
대규모 docs·위키·핸드북의 골격이다.

## 빠른 자가 점검 (어떤 글이든)

- [ ] 첫 문단이 **가치**(무엇을·왜)로 시작하는가, 아니면 배경·기능부터인가?
- [ ] 이 글의 **유형**이 분명하고 구조가 그 유형을 따르는가?
- [ ] 제목들이 키워드·평서문·일관 형식·30자 이내인가? H4 이상으로 깊지 않은가?
- [ ] 새 개념·단위·약어를 처음 등장 시 풀어줬는가?
- [ ] 문장이 능동·구체·한 생각인가? 번역체·한자어·모호어·메타담화가 없는가?
- [ ] 한마디로: **작가가 아는 순서**가 아니라 **독자가 알고 싶은 순서**인가?
