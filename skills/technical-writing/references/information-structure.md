# 정보 구조 만들기

**대전제:** "가장 중요한 건 문서를 *읽는 사람 입장*에서 설계하는 것." 잘 구조화된 정보는
작은 노력으로 이해된다. 구조 결함은 거의 다 "작가가 아는 순서대로 쓴" 증상이다.

각 원칙은 **규칙 → 이유 → before/after**로 적는다.

## ① 한 페이지에서는 하나만 다루기

- **규칙:** 한 페이지 = 하나의 핵심 목표. **제목 깊이가 H4(`####`) 이상이면 문서를 나누라는
  신호.** 개요 페이지를 두고, 주제는 최대한 구체적으로.
- **이유:** 가독성·탐색성 + 유지보수(변경 지점을 찾기 쉽고 중복 방지) + 구조 복잡도 억제.
- **Before:** `# React 사용법` → `## 컴포넌트 생성` → `### 기본 구조` → `#### JSX 사용법`
  → `##### JSX 문법` (주제가 너무 넓어 H5까지 내려감).
- **After:** `# React 컴포넌트 생성하기` → `## 기본 구조` / `## 상태 관리` / `## 라이프사이클`
  (H2 평면, 목표 하나, 깊이 얕음).

## ② 가치를 먼저 제공하기

- **규칙:** 기능·세부 설정보다 독자가 얻을 *가치*를 먼저. 배경·옵션은 나중에.
- **이유:** 독자는 "왜 필요한가, 내게 무슨 도움이 되나"를 먼저 알아야 끝까지 읽는다. 기능부터
  쓰는 것이 작가 본위의 대표 증상.
- **Before:** "리버스 프록시 설정은 2019년에 도입되었고…" / "이 스니펫은 `connection_timeout`,
  `retry_attempts` 등 다양한 옵션을 제공합니다."
- **After:** "리버스 프록시를 적용하면 **네트워크 지연을 최소화**할 수 있어요." / "이 스니펫을
  쓰면 **PostgreSQL 연결 속도가 50% 빨라집니다.**"

## ③ 효과적인 제목 쓰기

- **규칙:** ⓐ 핵심 키워드 포함 ⓑ 일관성(동사형/명사형 통일, 단계엔 숫자) ⓒ **30자 이내**
  ⓓ 평서문(`!`·`?` 금지).
- **이유:** 독자는 제목으로 스캔하고, 검색엔진은 제목으로 색인한다.
- **Before:** `# 에러를 해결하는 방법은?` → **After:** `` # `NOT_FOUND_USER` 에러를 해결하는 방법 ``.
- **Before(스타일 혼재):** `## 키워드를 포함하세요 / ## 일관성 유지 / ## 평서문으로 작성하기`
  → **After:** `## 키워드 포함하기 / ## 일관성 유지하기 / ## 평서문으로 작성하기`.

## ④ 개요 빠트리지 않기

- **규칙:** 제목 바로 아래·본문 시작 전에 핵심 요약. "이 문서를 읽으면 무엇을 할 수 있는가"에
  답한다. 기술 배경보다 핵심 정보를 먼저.
- **이유:** 독자가 빠르게 파악해 끝까지 읽을지 판단 + 검색 노출 + *작가도* 논점이 정리됨.
- **Before:** "이 문서는 TypeScript 유틸리티 타입을 소개합니다. Partial, Pick, Omit 등을 쓸
  수 있어요." (왜 쓰는지 없음)
- **After:** "**유틸리티 타입을 활용하면 반복적인 타입 선언을 줄이고 더 유연하게 타입을 관리**
  할 수 있어요. 이 문서에서는 Partial, Pick, Omit을…"

## ⑤ 예측 가능하게 하기

- **규칙:** 일관된 제목·형식·구조, 같은 수준은 같은 위계, 논리적 순서(기본 → 심화), 용어 일관.
- **이유:** 예측 가능한 구조는 인지 부담을 줄인다 — 독자가 새 문서를 접해도 기존 흐름을 연장해
  이해한다. 문서가 많아질수록 중요.
- **Before(설명 없이 코드 먼저):** 바로 `fetch(...)` → **After:** "**서버에서 데이터를 가져와
  JSON으로 변환하는 방법을 알아봅니다.**" 설명 후 코드.
- **Before(용어 혼용):** "상태(state)"·"데이터 저장"·"값 유지" → **After:** "상태(state)"·"상태
  저장"·"복잡한 상태 관리"로 통일.

## ⑥ 자세히 설명하기

- **규칙:** 새 개념은 한두 문장이라도 정의(필요하면 왜 알아야 하는지·어디 쓰는지). 동작은
  조건·입력·상태·단위 차이까지 충분히.
- **이유:** 필요한 정보를 생략하면 독자가 스스로 추론 → 오해·인지 부담.
- **Before:** "이 서비스는 이벤트 소싱 방식을 사용합니다." → **After:** "…이벤트 소싱(Event
  Sourcing)을 사용합니다. **이벤트 소싱은 최종 상태만 저장하는 대신, 상태를 바꾼 모든 이벤트를
  기록하는 방식입니다.**"
- **Before:** "`duration`: 세션 지속 시간." → **After:** 수동 로그아웃/시간 초과 종료가 각각
  무엇을 뜻하는지 + "**단위는 밀리초(ms)입니다.**"

> ⑥(자세히)과 문장 축의 '간결'은 모순이 아니다 — **필요한 정보는 빠짐없이(구조), 표현은 짧게
> (문장).** 정보량과 단어 수는 다르다.
