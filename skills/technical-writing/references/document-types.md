# 문서 유형 정하기

**왜 유형이 먼저인가:** 독자의 *목적*이 다르면 필요한 *정보의 형태*가 다르다. 유형은
"독자 목적의 분류"이고, 각 유형엔 검증된 표준 구조가 따라온다. 유형을 정하면 작가는 골격을
얻고, 독자는 필요한 정보를 빠르게 찾는다.

## 4대 유형 (독자 목적 → 핵심 → 표준 구조)

| 유형 | 독자 목적 | 핵심 가치 | 표준 구조 |
|---|---|---|---|
| **학습 중심** (시작하기·튜토리얼) | 새 기술을 처음 접해 흐름을 익힘 | 막힘없는 성공 경험 | 목표 → 사전 요구사항 → 단계별 가이드 → 결과 확인 |
| **문제 해결** (How-to·트러블슈팅) | 배경이 있는 상태에서 특정 문제 해결 | "지금 이 문제를 푸는가" | 문제 정의 → 사전 요구사항 → 해결 방법 → 확인 방법 |
| **참조** (Reference) | 작동법을 아는 상태에서 옵션을 빠르게 확인 | 정확성·완전성·검색성 | 개요 → 속성/옵션 → 시그니처 → 반환값 → 예제 |
| **설명** (Explanation) | 개념·원리·배경을 깊이 이해 | 배경·맥락·의사결정 과정 | 개념 → 등장 배경 → 작동 방식 → 활용 |

## 유형 선택 휴리스틱 (질문 → 유형)

- "this 키워드는 *어떻게 동작*하나?" → **설명** (원리·등장 배경)
- "HtmlWebpackPlugin 설정을 *어떻게 추가*하나?" → **How-to** (+ 옵션은 **참조**를 함께)
- "'Cannot read property of undefined'를 *어떻게 해결*하나?" → **트러블슈팅**
- "fs.readFile의 options를 *어떻게 쓰*나?" → **참조**

> 핵심 구분: 에러 코드 문서는 "이게 무슨 뜻이지?" 하고 찾는 **사전(참조)**, 트러블슈팅은
> "이걸 어떻게 해결하지?" 하고 보는 **해결 가이드**다.

## 유형별 세부 원칙

- **학습 중심:** *성공 경험*이 핵심. 모든 예제 코드는 실제로 실행·검증해 "따라하다 막히지
  않게". 점진적 난이도. 바로 쓸 수 있는 실행 가능 코드. 성공 경로에서 벗어난 부가 내용은
  FAQ·접이식으로 분리. (시작하기=흐름·개념, 튜토리얼=명확한 목표·결과물.)
- **문제 해결:** 문제 상황을 명확히 정의(원인과 현상을 *구분*, 에러 메시지·로그 포함). 바로
  적용 가능한 해결책 + 해결 *원리* 한 줄. 환경별 차이(OS/버전) 명시. (How-to=기능 *구현*,
  트러블슈팅=발생한 문제 *진단·디버깅*.)
- **참조:** 정확·완전·최신. 함수명 → 매개변수 → 반환값 → 예제 순으로 *표준화*(예측 가능).
  목차·앵커로 검색 가능. 사전 필수 정보(API 키·인증·헤더)는 **앞부분**에 — 독자가 항상
  순서대로 읽지 않으므로.
- **설명:** *등장 배경*으로 "왜 이 기술인가"를 설득. 다이어그램·흐름도 적극(설명 문서엔 시각
  자료가 핵심). 사용법을 넘어 원리·철학을 이해시킨다.

## 유형은 절대적이지 않다

"이 분류가 항상 정답은 아니에요." 더 효과적이면 **유형을 결합**한다(예: 같은 페이지에서
학습 + 설명). 단, 한 페이지의 *주된* 목적은 하나로 수렴시킨다(→ `information-structure.md`의
"한 페이지 하나의 목표").
