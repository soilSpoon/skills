# 문장 다듬기

목표: **한 문장이 한 생각을 명확히** 전한다. "필요한 정보는 빠짐없이(구조), 표현은 짧게(문장)."
기계적 치환 금지 — 각 규칙엔 예외가 있다.

## ① 주체를 분명하게 (능동태 + 주어 명확)

**대전제:** 기술 문서의 독자는 주로 개발자 — **읽는 사람이 곧 행동의 주체**다. 그래서 독자를
주어로 쓰는 능동태가 자연스럽다.

- **도구·시스템을 행동 주체로 쓰지 않는다.**
  - Before: "**이 라이브러리는** 데이터베이스 초기화를 수행해요." → After: "이 명령어를
    실행하면 데이터베이스를 초기화할 수 있어요."
  - **예외:** 도구/기술의 *동작 자체*를 설명할 땐 도구가 주어일 수 있다(예: "Impression
    로그는 두 가지 방식으로 동작해요"). 맹목적 치환 금지.
- **능동형을 쓴다.**
  - Before: "애플리케이션이 실행되기 전에 설정이 완료되어야 합니다." → After: "애플리케이션을
    실행하기 전에 설정을 완료하세요."

## ② 필요한 정보만 남기기 (간결 + 메타담화 최소화)

- **한 문장에 한 생각.** 긴 문장은 독자가 핵심을 놓치게 한다.
  - Before: "버튼을 클릭하면 다음 단계로 이동하게 되며, 그 이후의 작업을 진행할 수 있습니다."
    → After: "버튼을 클릭하면 다음 단계로 이동합니다. 이동 후 이어서 작업하세요."
- **메타담화(말에 대한 말)를 줄인다.** 3가지 유형: ① 흐름 설명("다음으로", "앞서 설명했듯이")
  ② 불필요한 강조("사실은", "결론적으로") ③ 독자 반응 유도("아시겠지만", "여러분도 아실").
  - Before: "앞에서 설명했지만, 결론은 이 설정을 변경하는 것이…" → After: "이 설정을 변경하는
    것이 가장 효과적입니다."

## ③ 구체적으로 쓰기 (가장 풍부한 챕터)

- **ⓐ 명사 대신 동사.** 동사 파생 명사("설정 수행", "검토 진행")는 의미가 흐릿하다.
  - Before: "코드 최적화 진행 후 배포 수행이 필요합니다." → After: "코드를 최적화한 후
    배포하세요."
- **ⓑ 모호한 표현 → 명확한 표현.** "가능성이 있다/일부 경우/필요할 수도"는 신뢰도를 깎는다.
  - Before: "기존 설정이 영향을 받을 수도 있습니다." → After: "기존 설정이 **삭제됩니다.**"
  - Before: "일부 브라우저에서 동작하지 않을 가능성이 있습니다." → After: "**Internet
    Explorer**에서는 동작하지 않습니다."
- **ⓒ 5W 누락 방지(누가·무엇을·어디에·어떻게).**
  - Before: "에러가 발생하면 로그를 확인하세요." → After: "에러가 발생하면 **애플리케이션
    서버의 에러 로그 파일(`/var/log/app/error.log`)을** 확인하세요."
- **ⓓ 업계 은어·직역투 경계.**
  - Before: "에러가 발생하면 Exception을 **던집니다**." (`throw`의 직역) → After: "에러가
    발생하면 **예외(`Exception`)를 일으킵니다.**"
- **ⓔ 구체적 수치·기준.**
  - Before: "데이터가 많을 때는 성능이 저하될 수 있습니다." → After: "데이터가 **10,000건을
    넘으면 응답 시간이 1초 이상** 걸립니다."
- **ⓕ 참조 맥락 설명.** 변수·설정값을 다시 쓸 때 어디서 정의했는지(앞 참조), 처음 언급 시
  어디서 쓸지(뒤 예고)를 링크로. (예: "이 키는 **3단계에서 인증에 사용됩니다.**")

## ④ 자연스러운 한국어 (한국어 고유)

- **불필요한 한자어 제거.** "수행하다/진행하다"는 빼도 의미가 전달된다.
  - Before: "로그 파일을 삭제하는 작업을 수행합니다." → After: "로그 파일을 삭제합니다."
- **번역체(영어 명사화) 교정.** 영어는 동사를 명사화(-tion/-ing)하지만 한국어는 동사 그대로가
  자연스럽다.
  - Before: "API 키를 이용한 사용자 인증 처리가 완료된 후, DB 접속 설정 진행이 가능합니다." →
    After: "API 키로 사용자를 인증한 후, DB에 접속하도록 설정할 수 있습니다."
  - Before: "이 API를 **통해** 데이터를 가져옵니다." → After: "이 API**로** 데이터를 가져옵니다."

## ⑤ 일관되게 쓰기

- **공식 기술 용어**(위키피디아·공식 문서 표기, 대소문자 유의). Before: "K8" / "자바스크립트"
  → After: "쿠버네티스(Kubernetes)" / "JavaScript".
- **같은 개념 = 같은 표현.** "추가/첨부/넣다"를 혼용 → "업로드"로 통일.
- **약어는 첫 등장 시 풀어쓰기.** Before: "이 기능은 SSR을 지원합니다." → After: "이 기능은
  SSR(Server-Side Rendering)을 지원합니다."
- **독자가 해석할 수 없는 참조 금지(자기완결성).** 약어와 같은 부류로, *내부 추적 참조*도
  독자 맥락에 의존한다. 로드맵 항목 번호("#10 후속")·이슈/티켓 ID·"이전 PR"·"지난 회의에서"
  처럼 작가만 아는 포인터는, 미래의·외부의 독자가 풀 수 없다. 그것이 *무엇인지*를 본문에
  풀어 자기완결적으로 쓴다. Before: "프로젝트 메모리(#10) 후속, #2161 위에." → After:
  "프로젝트 기본값을 기억하는 기능에 이어, 그 저장·주입 위에 UI를 얹는다." (의존 관계는
  번호가 아니라 *그게 무엇을 하는지*로 서술.)
- **외래어 표기는 빈도 고려(토스 규칙).** 구글 트렌드 기준 외래어 표기법보다 5배 이상 많이
  쓰이는 표기가 있으면 그 표기를 사용. 예: "프런트엔드"(맞춤법) → "프론트엔드"(실사용).
