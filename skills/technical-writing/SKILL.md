# Technical Writing — 독자가 목표를 이루게 하는 기술 글쓰기

**좋은 기술 글 = 독자가 더 빠르고 정확하게 목표를 이루는 글.** 토스 테크니컬 라이팅
가이드(technical-writing.dev / `toss/technical-writing`)를 스킬로 옮긴 것이다. 일반
글쓰기는 감정·관점을 전하지만, 테크니컬 라이팅은 **정보를 빠르고 정확하게 전달**하는 데
집중한다. 작가의 자기표현·완결성·우아함은 가치가 아니다 — 판정 기준은 오직 "독자가 더
빠르고 정확하게 목표를 이루는가"다.

## 왜 — 모든 규칙의 단 하나의 뿌리

독자는 **인지 자원이 한정된 정보 처리자**다. 작가가 1의 노력을 아끼면, 수백 명의 독자가
각자 *추론·오적용·재방문*의 비용을 치른다. 이 **오해의 비용(cost of misunderstanding)**이
모든 규칙을 정당화한다. 그래서 항상 **작가의 편의 < 독자의 인지 절약**.

> 거의 모든 나쁜 글은 한 가지 증상이다 — **작가 본위로 썼다.** 내가 아는 순서대로,
> 내가 말하고 싶은 것 위주로 구조화한 것. 개요를 빠뜨리고, 가치보다 기능을 먼저 쓰고,
> 배경부터 늘어놓는 것 전부 이 증상이다. **교정의 핵심은 "독자 본위로 뒤집기".**

## 3축 — "글쓰기 실패의 3가지 원인을 뒤집은 것"

| 축 | 실패 원인 | 핵심 질문 | reference |
|---|---|---|---|
| **문서 유형** | 유형을 모르고 씀 | 독자의 *목적*에 맞는 형태인가? | `references/document-types.md` |
| **정보 구조** | 효과적으로 구조화 못 함 | 독자가 작은 노력으로 이해하는가? | `references/information-structure.md` |
| **문장** | 이해하기 좋은 문장이 어려움 | 한 문장이 한 생각을 명확히 전하는가? | `references/sentence-craft.md` |

## 두 가지 모드

이 스킬은 두 상황에서 쓴다. **단계를 섞지 말 것** — 가이드 자신이 "유형 → 구조 → 문장을
별도 봇으로 나눠야 답이 잘 나온다"고 못 박는다.

1. **생성(write)**: ① 문서 유형 분류(`document-types.md`의 결정 트리) → ② 그 유형의 표준
   골격 → ③ 정보 구조·문장 원칙 적용.
2. **리뷰/개선(review)**: `references/review-checklist.md`의 3단계 체크리스트로 *진단* →
   각 reference의 **before/after 패턴**으로 *통합된 단일 개선안* 생성(여러 안 나열 금지).

## 트리거 맵 — 패턴 → reference

| 신호 | 펴 볼 reference |
|---|---|
| "무슨 문서를 어떻게?" · 튜토리얼/How-to/참조/설명 혼동 | `document-types.md` |
| 개요 없음 · 기능부터 시작 · 제목 모호 · H4 이상 · 위계 불규칙 · 용어 혼용 | `information-structure.md` |
| 수동태 · 도구가 주어 · 긴 문장 · "수행/진행" · 모호어 · 메타담화 · 번역체 · 약어 미해설 | `sentence-craft.md` |
| "이 글 리뷰/개선해줘" | `review-checklist.md` (먼저) → 해당 축 reference |
| PR 설명 · 설계 문서 · README · 커밋 메시지 · 릴리스 노트 | `applied-contexts.md` |

> ⚠️ **로드 규율** — `references/*.md`는 **필요할 때만** 1–2개씩 연다. 짧은 글(커밋·PR
> 한 단락)은 본문 + 트리거 맵으로 충분하다.

## 항상 지키는 바닥

- **독자 본위로 뒤집기**가 1번. 막히면 "독자가 이 시점에 무엇을 알고 싶은가?"를 묻는다.
- **정보는 빠짐없이, 표현은 짧게.** 정보량과 단어 수는 다르다 — 필요한 배경은 다 넣되
  (`information-structure.md`의 "자세히 설명"), 문장은 군더더기를 깎는다(`sentence-craft.md`).
- **규칙은 교조가 아니라 휴리스틱.** 가이드 자신이 "권장이며 맥락에 따라 유연하게"라고
  한다. 예외를 존중한다(예: 도구의 *동작 자체*를 설명할 땐 도구가 주어일 수 있다). 기계적
  치환 금지.
- **한국어 고유 규칙**(동사 우선·한자어 제거·번역체 교정·외래어 빈도)은 한국어 글에서
  반드시 적용 — `sentence-craft.md` 참조.

## code-fundamentals와의 관계

`code-fundamentals`는 **코드를 읽는** 비용을, 이 스킬은 **글(문서·PR·주석)을 읽는** 비용을
줄인다. 인접한 두 fundamentals다. 코드 변경에 딸린 PR 설명·설계 문서·README·주석에는
**둘을 함께** 쓴다 — 코드는 4축으로, 그 곁의 산문은 이 3축으로.
