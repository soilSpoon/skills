# 이력서/경력기술서 가이드라인

모든 피드백과 규칙을 통합한 단일 참조 문서.
이력서/경력기술서 작성 또는 수정 시 반드시 이 문서를 따른다.

---

## 1. 핵심 어필 전략

### 서사 구조: "비효율 → 효율화"

경력기술서의 모든 항목은 이 흐름을 따라야 한다:
1. 회사/프로그램이 **비효율적으로 돌아가던 상황**
2. 내가 **불편함을 느끼고 개선 방법을 고민**
3. **구체적으로 어떻게 바꿨고**, 어떤 **업무 효율 증진/시간 단축**을 이뤘는지

### 어필해야 할 3가지 강점

1. **비효율 해결사** — 비효율을 보면 더 효율적으로 바꿀 수 있는 것은 없는지 고민하고 해결하는 것이 핵심 강점. 오픈소스 기여(CLI 사용 중 버그 발견 → 직접 수정)도 이 맥락 — 취미로도 할 만큼 불편함/비효율을 보면 해결 방법을 찾는 사람.

2. **빠른 기술 적응력** — 출근노트에서 새로운 툴을 워크플로우에 들여온 사례들 어필. 빠르게 바뀌는 개발 문화에 가장 빨리 적응할 수 있는 사람임을 보여줘야 한다.

3. **AI 시대 적합 인재** — 새로운 기술이 나오면 가장 먼저 써보고 관심을 가지며, 그 기술을 워크플로우에 어떻게 들여와서 효율성을 높일지 고민하는 사람. AI 시대의 개발자로 적합한 인재임을 어필.

## 2. 제목 프레이밍

**잘하는 것(역량)이 제목이어야 한다. 잘했던 것(작업) 나열이 아님.**

- **Bad** (작업 나열): "C++ CAD 엔진의 Emscripten WASM 컴파일 및 Next.js SSR 브라우저 이식"
- **Good** (역량 영역): "React, TypeScript, Next.js 기반 프로덕트 아키텍처 설계 — jQuery 레거시 전환부터 Chakra UI 53개 공통 컴포넌트 체계까지"

### 확정된 제목 예시 (6개)

1. React, TypeScript, Next.js 기반 프로덕트 아키텍처 설계 — jQuery 레거시 전환부터 Chakra UI 53개 공통 컴포넌트 체계까지
2. React-Query·Zustand·Jotai 3계층 상태 관리 설계 및 react-window + Redis 양단 성능 최적화
3. B2B2C 5단계 권한 체계 설계 및 어드민·노무법인 시스템 6개 모듈 신규 개발
4. PostHog 이벤트 추적 파이프라인 구축으로 운영 대시보드 기반 데이터 의사결정 체계 수립
5. AI SDK v5 챗봇으로 CAE 시뮬레이션 5단계 워크플로우 자동화 — 운영 효율성을 위한 자연어 인터페이스 설계
6. Webpack·SWC·Babel 빌드 체계 소스 레벨 수정 및 오픈소스 코어 기여 (laravel-mix 12 PRs, Goose, CLIProxyAPI)

## 3. 상단 요약문

**확정안:**
> 사용자 업무 프로세스를 이해하고 비효율을 구조적으로 제거하는 데 집중합니다. 새로운 기술을 프로덕트에 가장 먼저 도입하되, 표면 적용에 그치지 않고 소스 레벨까지 파고들어 근본을 개선합니다.

## 4. 디자인 규칙 (CareerLayout/ResumeLayout)

### 시각적 위계

| 요소 | 현재 문제 | 개선 |
|------|---------|------|
| 제목 (회사명/프로젝트명) | 제목처럼 안 보임 | 크기 키우고 시각적 구분 강화 |
| 소제목 (이슈/개선/결과) | 눈에 안 띔 | 색상·크기를 더 눈에 띄게 |
| 본문 | 너무 흐릿함 | 검은색으로 통일 |
| 구분선 | 너무 옅음 | 더 진하게 |
| 위계 전체 | 중요/비중요 색차이 없음 | 제목·소제목·본문·메타 간 명확한 위계 |

### 간격/여백

| 규칙 | 설명 |
|------|------|
| 전반적으로 축소 | 섹션 간, 항목 간, 행간 모두 줄이기 |
| 들여쓰기 통일 | 할 건지 말 건지 하나로 통일, 너무 깊지 않게 |
| 회사 전환 구간 | 확실히 끊어지는 것이 보이게 |
| 새 회사 배치 | 여백 조절해서 페이지 초입에 오면 BEST |

### Bold 사용

| 영역 | 규칙 |
|------|------|
| 개선 사항 | Bold **전부 제거**. 내가 뭘 했는지는 텍스트로 자연스럽게 |
| 적용 결과 | **여기서만** 성과/수치를 Bold로 강조 |
| 동사 | "~을 구현하여", "~을 도입하고" 등에 Bold **금지** |
| 핵심만 | "개발 시간 50% 단축", "UI 프리징 해소" 같은 핵심 성과만 Bold |

**Good**: "React 도입을 **건의**하고" (주도성), "**4년간 안정적으로 확장 가능한 프론트엔드 기반** 확보"
**Bad**: 개선 사항 전체 Bold, 상황 설명에 Bold

## 6. 내용 구조

| 규칙 | 근거 |
|------|------|
| 상황 설명 NO | "이건 PPT에 통하는 것임, 절대 금지." 상황 설명은 면접에서 |
| 내가 한 것 어필 중심 | 제목 체계: 역량 영역 → 세부 성과 |
| 중요한 것 문두에 | 첫 줄에서 핵심이 보여야 함 |
| 문장 끝 표시 | 불릿이 길면 어디가 문장 끝인지 명시 |

## 적용 대상

### 데이터 (master.yaml)
- 요약문 교체 (summaries.default)
- 경력기술서 제목 리프레이밍 (details[].title)
- details[].content 내 Bold 마크다운 수정
- 문체 어미 통일
- 오픈소스 섹션 구조 재편
- "비효율 → 효율화" 서사 강화

### 레이아웃 (CareerLayout.svelte)
- 제목 크기/색상 위계 강화
- 소제목(이슈 사항, 개선 사항 등) 스타일 조정
- 간격/행간 축소
- 구분선 색상 진하게
- 회사 전환 구간 시각적 구분 강화
- 본문 색상 검은색 통일

### 이력서 (ResumeLayout.svelte)
- 가독성/UI 미세 조정
- 동일한 간격/Bold 규칙 적용
