### 9개 이슈 제목 (익명화, 구조 패턴만)

1. **컨테이너 오케스트레이션 스팟 인스턴스 재생성 이슈**
   - 이슈: 스팟 인스턴스 재생성 시 Pod Rollout 실패
   - 문제점: 컨트롤러 로직 수준의 원인 분석 (코드 레벨)
   - 개선: 컨트롤러 로직 수정 + 운영 환경 정책 변경
   - 결과: 다운타임 발생 문제 해결

2. **VM → 컨테이너 무중단 마이그레이션**
   - 이슈: 자원 비효율, 배포 시간 지연, 배포 방식 제한
   - 의사 결정: 브랜치 전략 + 빌드/배포 전략 비교 검토
   - 실행 방안: Blue-Green 도메인 이전 vs 가중치 기반 점진적 이관
   - 결과: **배포 시간 10분 → 2분으로 단축**, 무중단 마이그레이션 완료

3. **DNS 리졸빙 장애 원인 분석**
   - 이슈: 클러스터 업그레이드 후 특정 앱에서만 DNS 실패
   - 문제점: kube-proxy known issue + 프레임워크 소켓 특성 분석
   - 개선: 3가지 컴포넌트(kube-proxy, CoreDNS, 프레임워크) 각각 대응
   - 결과: DNS 리졸빙 실패 재발 방지

4. **GitOps 도구에 Helmfile 통합**
   - 이슈: 로컬-운영 환경 코드 분리 + 관리 포인트 증가
   - 개선: CMP 플러그인 도입으로 Helmfile 직접 렌더링
   - 결과: 개발/운영 환경 간 코드 일관성 확보

5. **GitOps 도구 성능 개선**
   - 이슈: MonoRepo 환경에서 성능 저하 (불필요한 Manifest 재생성)
   - 문제점: 3분 주기 Polling + 저장소 크기 + 중복 다운로드
   - 개선: Webhook 기반 변경 감지 + tar 용량 최적화 + 캐시 재사용
   - 결과: **CPU 사용률 12 Core → 0.4 Core**, **Manifest 재생성 7,000회 → 1,000회/주**

6. **서비스 메시 도입으로 로드밸런서 통합 및 비용 절감**
   - 이슈: API당 로드밸런서 개별 생성 → 비용 증가 + 보안 그룹 중복
   - 개선: Istio Ingress Gateway로 통합
   - 결과: **500개 이상 → 약 50개로 통합**, **월 비용 $1,000 → $70**

7. **GPU 노드 이벤트 기반 스케일링 아키텍처 설계**
   - 이슈: GPU 노드 상시 운영 시 비용 낭비
   - 실행: KEDA + Karpenter + DLQ 조합
   - 결과: 작업 상황에 맞춘 동적 운영, 비용 절감

8. **IaC 도구를 활용한 클라우드 리소스 관리**
   - 이슈: 서비스마다 동일 리소스 수동 생성 반복
   - 개선: 모듈화 + 환경별 디렉토리 + State 분리 + PR 기반 적용
   - 결과: 리소스 재사용성 향상, 변경 이력 가시성 개선

9. **제로트러스트 기반 클러스터 접근 권한 관리**
   - 이슈: IAM 기반 접근 관리의 복잡성 + 감사 어려움
   - 의사 결정: 내부회계 관리 제도(ITGC) 감사 대응 + 세션 녹화 필요
   - 개선: Teleport 도입, GitHub SSO 통합, 세션 녹화
   - 결과: 접근 권한 중앙 관리, 감사 요구사항 충족

## 이 벤치마크에서 배울 점

1. **제목이 문제 해결 영역**: "Karpenter 스팟 인스턴스 재생성 이슈" — 작업이 아닌 도메인
2. **코드/로직 수준 분석**: 단순 "이런 문제가 있었다"가 아니라 컨트롤러 로직, 네트워크 계층까지 파고든 과정
3. **의사 결정 사항 섹션**: 여러 방안을 비교하고 왜 그 방법을 선택했는지 기술
4. **수치 기반 성과**: "12 Core → 0.4 Core", "$1,000 → $70", "7,000회 → 1,000회"
5. **적용 결과에서만 Bold**: 개선 사항은 일반 텍스트, 결과의 핵심 수치만 강조
