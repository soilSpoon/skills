# 지원서 작성 규칙

## 템플릿 (4단 구조)

```
안녕하세요. React/TypeScript 7년차 풀스택 개발자입니다.

[프로젝트 분석]
본 프로젝트의 핵심은 다음과 같이 파악됩니다:
1. (핵심 1)
2. (핵심 2)
3. (핵심 3)

예상되는 주요 이슈:
- (이슈 1): (왜 이슈인지 한 줄 설명)
- (이슈 2): (왜 이슈인지 한 줄 설명)

[유사 프로젝트 경험]
1) (경험 제목 — 익명화)
- (프로젝트 배경 한 줄)
- 실제로 겪은 이슈: (XX 문제) → (YY 방식으로 해결)
- 결과: (성과/효과)

2) (경험 제목 — 익명화)
- (프로젝트 배경 한 줄)
- 실제로 겪은 이슈: (XX 문제) → (YY 방식으로 해결)
- 결과: (성과/효과)

[진행 프로세스]
1단계. (단계명) — (확정 기간)
- (구체적 내용)
- (산출물/공유 항목)

2단계. (단계명) — (확정 기간)
- (구체적 내용)
- (중간 데모/공유 포함)

3단계. (단계명) — (확정 기간)
- (구체적 내용)

[미팅 시 논의 희망 사항]
- (논의 사항 1)
- (논의 사항 2)

포트폴리오와 상세 경험은 프로필에서 확인 가능합니다.
감사합니다.
```

## DO

- 프로젝트별 맞춤 작성 — 복붙 금지, 프로젝트 고유 내용 필수
- 경험 중심 어필 — 기술 디테일보다 "무엇을 해서 어떤 결과를 냈는지"
- 개조식/명사형 종결 — "~했습니다" 반복보다 "~수행", "~해결"
- 830자 이상 (위시켓 데이터봇 권장)
- 포트폴리오 2-3개 첨부 (매칭 확률 60% 상승)
- 모든 경험에 "이슈→해결→결과" 패턴 필수 (나열 금지)
- "미팅 시 논의 희망 사항" 포함
- 기간은 확정형 ("2주") 사용

## DON'T

- 기술 스택 나열 섹션
- 컴포넌트/훅 이름 (PermissionGate, usePermission 등)
- 기술 버전 (AI SDK v5 등)
- 코드량 지표 (커밋 수, PR 수)
- 범위형 기간 ("1~2주", "2~3주")
- 기술 용어 직접 사용 — 비즈니스 언어로 전환:

| 금지 | 대체 |
|------|------|
| hospital_id, tenant_id | 병원별/테넌트별 데이터 격리 |
| RBAC | 역할별 접근 권한 체계 |
| Chart.js, D3 | 시각화 대시보드 |
| API 레이어 추상화 | 데이터 호출 구조 정리 |
| WebSocket | 실시간 데이터 연동 |
| 가상화 (virtualization) | 대량 데이터 화면 최적화 |
| React Query, Zustand, Jotai | (언급하지 않음) |

## 금액/기간 산정

단순 "예산 동일"이 아닌 4단계 분석:

**Step 1: 스코프 분석** — 기능 모듈 수, 업무 범위(개발/디자인/기획), MVP vs 풀 스코프
**Step 2: 공수 추정** — 모듈 × 복잡도 = 인일(man-day)
**Step 3: 일당 단가** — 예산÷기간. 시장 기준 시니어 풀스택 25~40만/일
**Step 4: 최종 결정**
- feasibility 4.0+: 예산 동일 또는 기간 단축(-10%)
- feasibility 3.0~3.9: 예산 동일, 기간 동일
- feasibility 2.0~2.9: MVP 범위 한정 명시, 예산 동일 (저가 경쟁 금지)
- 일당 25만 미만: 스코프 조율 필요 멘트 추가
- 일당 40만 이상: 기간 단축으로 가치 어필

**핵심 원칙:**
- 저가 경쟁 금지 (위시켓 데이터: 싸게 써서 계약 14%뿐)
- 첫 수주는 확실한 프로젝트에서 5.0 리뷰 확보가 목표
