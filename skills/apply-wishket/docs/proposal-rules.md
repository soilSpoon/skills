## 템플릿 (자연스러운 흐름)

**핵심 원칙: AI가 쓴 티를 내지 않는다.** `[대괄호 섹션]`, "본 프로젝트의 핵심은 다음과 같이 파악됩니다:" 같은 정형화된 구조를 쓰지 않는다. 지원자 61명 중 상당수가 AI 템플릿을 쓰므로, 자연스러운 문체가 차별화 요소다.

**프로젝트 유형에 따라 어조를 맞춘다:**
- 퍼블리싱 → 컴포넌트 완성도, Figma 변환 속도, 반응형 등 퍼블리싱 언어
- SaaS/플랫폼 → 아키텍처, 확장성, 권한 체계 등 시스템 언어
- MVP → 빠른 검증, 최소 기능, 피봇 가능한 구조 등 린 스타트업 언어

```
안녕하세요. React/TypeScript 기반 풀스택 개발자입니다.

(프로젝트 핵심 니즈에 공감하는 1-2문장 — 프로젝트 설명을 복붙하지 말고, 클라이언트가 풀려는 문제를 자기 언어로 요약)

이 프로젝트에서 까다로울 수 있는 부분은 (이슈 1)과 (이슈 2)인데, 비슷한 상황을 직접 겪어본 경험이 있습니다.

(경험 1 — 자연스러운 문단으로. "실제로 겪은 이슈:" 같은 라벨 없이)
(프로젝트 배경) → (겪은 문제) → (해결 방식) → (결과)

(경험 2 — 마찬가지로 자연스럽게)

진행은 이렇게 생각하고 있습니다:
- 1단계. (단계명) — (확정 기간): (핵심 내용)
- 2단계. (단계명) — (확정 기간): (핵심 내용), 중간 데모
- 3단계. (단계명) — (확정 기간): (핵심 내용)
주 2회 진행 상황을 공유드리겠습니다.

미팅에서 (논의 사항 1)과 (논의 사항 2)를 여쭤보고 싶습니다.
(클라이언트가 미처 고려하지 못했을 포인트 1개 — 제안 형태)

포트폴리오와 상세 경험은 프로필에서 확인 가능합니다.
감사합니다.
```

**변형 허용**: 위 구조는 가이드라인이지 강제가 아니다. 프로젝트에 따라 경험을 먼저 쓰거나, 프로세스를 생략하거나, 논의 사항을 확대할 수 있다. 핵심은 **자연스러운 흐름**과 **프로젝트에 맞는 어조**.

## DO

- 프로젝트별 맞춤 작성 — 복붙 금지, 프로젝트 고유 내용 필수
- 경험 중심 어필 — 기술 디테일보다 "무엇을 해서 어떤 결과를 냈는지"
- 개조식/명사형 종결 — "~했습니다" 반복보다 "~수행", "~해결"
- 830자 이상 (위시켓 데이터봇 권장)
- 포트폴리오 2-3개 첨부 (매칭 확률 60% 상승)
- 모든 경험에 "이슈→해결→결과" 패턴 필수 (나열 금지)
- "미팅 시 논의 희망 사항" 포함
- 기간은 확정형 ("2주") 사용
- **진행 프로세스에 중간 공유 포인트 명시** — "주 1회 진행 공유", "중간 데모 N회" 등 (성공 파트너 평가에서 "소통", "적극성"이 반복 언급)
- **클라이언트가 미처 생각하지 못한 포인트 1개 제안** — "논의 사항"에 포함 (성공 파트너 평가: "생각하지 못한 부분까지도 제안")

## DON'T

- 기술 스택 나열 섹션
- 컴포넌트/훅 이름 (PermissionGate, usePermission 등)
- 기술 버전 (AI SDK v5 등)
- 코드량 지표 (커밋 수, PR 수)
- 범위형 기간 ("1~2주", "2~3주")
- 기술 용어 직접 사용 — 비즈니스 언어로 전환:

| 금지 | 대체 |
|------|------|
| hospital_id, tenant_id | 병원별/테넌트별 데이터 격리 |
| RBAC | 역할별 접근 권한 체계 |
| Chart.js, D3 | 시각화 대시보드 |
| API 레이어 추상화 | 데이터 호출 구조 정리 |
| WebSocket | 실시간 데이터 연동 |
| 가상화 (virtualization) | 대량 데이터 화면 최적화 |
| React Query, Zustand, Jotai | (언급하지 않음) |

## 금액/기간 산정

6단계 분석. 단순히 "예산 동일"이 아니라, 클라이언트 예산의 현실성을 먼저 판단하고, 제안 금액이 예상에서 크게 벗어나지 않도록 조율한다.

### Step 1: 스코프 분석

프로젝트 설명에서:
- 기능 모듈 수 (인증, CRUD, 대시보드, 관리자, 결제 등)
- 업무 범위 (개발만 vs 개발+디자인+기획)
- MVP인지 풀 스코프인지
- 디자인 시안 유무 (있으면 공수 절감)

### Step 2: 공수 추정

모듈 × 복잡도로 인일(man-day) 산정. 참고 기준:

| 작업 유형 | 인일 기준 |
|----------|----------|
| 기획/설계 (ERD, 화면설계) | 3~7일 |
| 인증/권한 모듈 | 3~5일 |
| CRUD 1세트 (목록+상세+폼) | 2~3일 |
| 대시보드/차트 | 3~5일 |
| 결제 연동 (PG) | 3~5일 |
| 관리자 페이지 | 5~10일 |
| 모바일 앱 (RN) | 10~20일 |
| QA/수정/배포 | 전체의 15~20% |

### Step 3: 일정 강도(Intensity) 분석

기간이 빡빡할수록 하루에 더 많은 시간을 투입해야 하므로, 일당을 더 높게 받아야 한다. 이를 정량화한다.

```
가용 작업일 = 기간(캘린더일) × 5/7   (주 5일 기준, 소수점 내림)
강도 = 추정 공수(인일) ÷ 가용 작업일
```

| 강도 | 판단 | 의미 | 일당 프리미엄 |
|------|------|------|-------------|
| < 0.6 | 여유 | 파트타임도 가능 | 없음 (기간 단축 제안 가능) |
| 0.6~0.8 | 보통 | 풀타임, 적정 속도 | 없음 |
| 0.8~1.0 | 타이트 | 풀타임 + 버퍼 없음 | **+10~20%** |
| 1.0~1.3 | 매우 타이트 | 야근/주말 필요 | **+20~30%** |
| > 1.3 | 불가능 | 1인으로 물리적 불가 | 스코프 축소 필수, 또는 지원 재고 |

예시:
- 공수 12인일, 기간 15일 → 가용 10일 → 강도 1.2 → "매우 타이트" → 프리미엄 +20~30%
- 공수 74인일, 기간 90일 → 가용 64일 → 강도 1.16 → "매우 타이트" → 프리미엄 +20~30%
- 공수 50인일, 기간 50일 → 가용 36일 → 강도 1.39 → "불가능" → 스코프 축소 필수

강도가 높은 프로젝트는 **일당을 낮추면 안 된다** — 오히려 높여야 한다. 빡빡한 일정에 싼 가격은 "대충 하겠다"는 인상을 준다.

### Step 4: 클라이언트 예산 현실성 판단

위시켓 프로젝트에는 두 가지 금액이 있다:
- **예상 금액**: 클라이언트가 설정한 예산
- **예상 기간**: 클라이언트가 설정한 기간

이 둘로 **클라이언트 기대 일당**을 계산한다:

```
클라이언트 기대 일당 = 예상 금액 ÷ 예상 기간
```

**강도 대비 예산 현실성**을 판단한다 — 단순 일당뿐 아니라 강도를 반영:

```
강도 보정 일당 = 클라이언트 기대 일당 ÷ 강도
  (강도 1.0이면 동일, 1.2면 실질 일당이 83%로 떨어짐)
```

| 강도 보정 일당 | 판단 | 대응 |
|--------------|------|------|
| 15만 미만 | 비현실적 | 지원 재고. 스코프 대폭 축소 필요 |
| 15~25만 | 타이트 | MVP 범위 한정 또는 기능 우선순위 조율 |
| 25~40만 | 적정 | 기본 시나리오 |
| 40만+ | 여유 | 기간 단축 또는 품질 강화로 가치 어필 |

### Step 5: 내 제안 금액/기간 결정

**제안 금액**과 **제안 기간**을 결정한다. Step 3(강도)과 Step 4(예산 현실성)를 모두 반영.

**일당 하한선: 10만원/일** — 이보다 낮게 제안하지 않는다. 신규 파트너는 10~20만원/일 범위, BOOST 75% 이하.

**강도 프리미엄 적용 공식:**

```
기본 제안 일당 = 클라이언트 기대 일당 (예산÷기간)
강도 프리미엄 = Step 3의 프리미엄 (0%, +10~20%, +20~30%)
제안 일당 = 기본 일당 × (1 + 강도 프리미엄)
제안 금액 = 제안 일당 × 제안 기간
```

단, **이탈도 20% 초과하면 안 되므로** 프리미엄은 이탈도 한도 내에서 조정.

| 강도 | 예산 현실성 | 전략 |
|------|-----------|------|
| 여유 (<0.6) + 적정 예산 | 기간 단축 제안 → 일당 자연 상승 | "N일 앞당겨 납품 가능" |
| 보통 (0.6~0.8) + 적정 예산 | 예산 동일, 기간 동일 | 기본 시나리오 |
| 타이트 (0.8~1.0) + 적정 예산 | 예산 동일, 기간 동일 | 지원서에 "집중 투입" 뉘앙스 |
| 매우 타이트 (1.0~1.3) + 적정 예산 | **예산 +10~20%** 또는 기간 +10~20% | 프리미엄 근거: 집중 투입, 야근/주말 |
| 매우 타이트 + 타이트 예산 | 예산 동일 + **스코프 축소 필수 멘트** | "이 기간에 전체 구현은 무리, MVP 우선" |
| 불가능 (>1.3) | 예산 동일 + **MVP만 명시** + 2차 분리 | "전체 스코프는 N일 필요, 예산 내 MVP 제안" |

**핵심: 빡빡한 일정에 싼 가격을 제안하지 않는다.** 강도가 높을수록 일당이 올라가는 것이 정상이다. 클라이언트도 빡빡한 일정을 알고 있으므로, 합리적 프리미엄은 오히려 전문성 인상을 준다.

### Step 6: 이탈도 체크

제안 금액/기간이 클라이언트 예상에서 **과도하게 벗어나면** 미팅 성사 확률이 낮아진다.

```
금액 이탈도 = |제안 금액 - 예상 금액| ÷ 예상 금액 × 100%
기간 이탈도 = |제안 기간 - 예상 기간| ÷ 예상 기간 × 100%
```

| 이탈도 | 판단 | 조치 |
|--------|------|------|
| 0~10% | 안전 | 그대로 진행 |
| 10~20% | 주의 | 지원서에서 이유를 자연스럽게 설명 (스코프 조율, 기간 단축 근거) |
| 20%+ | 위험 | 재조정 필요. 20% 이상 벗어나면 클라이언트가 검토조차 안 할 가능성 |

예외: 기간을 10~15% 단축하는 것은 긍정적 이탈 (빠른 납기 = 가치). 금액을 20%+ 높이는 것은 부정적 이탈.

### Step 7: 최종 확인 + 벤치마크 비교

**내 제안 일당** = 제안 금액 ÷ 제안 기간

이 값을 벤치마크와 비교:

| 기준 | 일당 범위 | 근거 |
|------|----------|------|
| 시장 시니어 풀스택 | 25~40만/일 | 위시켓 일반 시세 |
| 성공 파트너 A (12건, 5.0) | 25~30만/일 | 150만/6일 ~ 3000만/99일 |
| 성공 파트너 B (31건, 5.0) | 9~23만/일 | 80만/9일 ~ 850만/70일 (볼륨형) |
| 성공 파트너 C (78건, 4.93) | 다양 | 60만 ~ 1000만 (니치 도메인) |
| **일당 하한선** | **10만/일** | 이보다 낮게 제안하지 않음 |
| **신규 파트너 현실 범위 (리뷰 0건)** | **10~20만/일** | 첫 수주 확보가 최우선. BOOST 75% 이하 + 이 범위 내에서 결정 |
| **리뷰 1건+ 파트너 범위** | **15~33만/일** | 리뷰 확보 후 단가 상향 |

핵심 인사이트:
- 성공 파트너도 초기에는 소규모(150만/6일, 25만/일)로 시작
- **"미팅 성사 파트너 금액이 평균보다 높았다"는 리뷰가 있는 파트너 기준** — 신규 파트너(리뷰 0건)에는 해당 안 됨
- 신규 파트너는 **가격 경쟁력이 핵심 차별화 요소** — 경력과 포트폴리오가 아무리 좋아도 실적 0건이면 검증된 파트너에 밀림
- **첫 수주 전략**: 소규모에서 5.0 리뷰 확보 → 이후 단가 상향

### BOOST 금액 통계 활용 (필수)

Phase 1에서 수집한 BOOST 금액 통계(최저/75%/평균/25%/최고)를 금액 결정에 반영한다.

**신규 파트너 금액 결정 규칙:**
1. **BOOST 75% 구간 이하**로 책정 — 중간값 이상은 신규 파트너에게 불리
2. **일당 10~20만원 범위** 유지 (첫 수주 확보 전까지)
3. 클라이언트 과거 계약 일당이 있으면 그 범위에 맞춤

**Example:**
```
BOOST 통계: 최저 220만, 75% 490만, 평균 470만, 최고 700만
→ 신규 파트너 제안: 350~490만 (75% 이하)
→ 일당 계산: 350만/15일=23만 (범위 초과) → 300만/15일=20만 (범위 내) ✓
```

리뷰 1건 이상 확보 후에는 75% 구간까지 올릴 수 있다.

### 출력 형식 (작성 근거에 포함)

```
### 금액/기간 산정
- 스코프: {모듈 수, 범위, MVP 여부}
- 공수 추정: {N}인일 (근거: 모듈별 인일 산정)
- 강도: {N}인일 ÷ 가용 {M}일 = {강도} → {여유/보통/타이트/매우 타이트/불가능}
- 클라이언트 기대 일당: {예상 금액}÷{예상 기간} = {N}만원/일
- 강도 보정 일당: {기대 일당}÷{강도} = {N}만원/일 → {적정/타이트/여유/비현실적}
- 강도 프리미엄: {0%/+10~20%/+20~30%}
```
