# 지원서 작성 규칙

## 템플릿 (4단 구조)

```
안녕하세요. React/TypeScript 7년차 풀스택 개발자입니다.

[프로젝트 분석]
본 프로젝트의 핵심은 다음과 같이 파악됩니다:
1. (핵심 1)
2. (핵심 2)
3. (핵심 3)

예상되는 주요 이슈:
- (이슈 1): (왜 이슈인지 한 줄 설명)
- (이슈 2): (왜 이슈인지 한 줄 설명)

[유사 프로젝트 경험]
1) (경험 제목 — 익명화)
- (프로젝트 배경 한 줄)
- 실제로 겪은 이슈: (XX 문제) → (YY 방식으로 해결)
- 결과: (성과/효과)

2) (경험 제목 — 익명화)
- (프로젝트 배경 한 줄)
- 실제로 겪은 이슈: (XX 문제) → (YY 방식으로 해결)
- 결과: (성과/효과)

[진행 프로세스]
1단계. (단계명) — (확정 기간)
- (구체적 내용)
- (산출물/공유 항목)
- (중간 공유: 주 N회 진행 보고 / 중간 데모)

2단계. (단계명) — (확정 기간)
- (구체적 내용)
- (중간 데모/공유 포함)

3단계. (단계명) — (확정 기간)
- (구체적 내용)
- (최종 산출물)

[미팅 시 논의 희망 사항]
- (논의 사항 1)
- (논의 사항 2)
- (클라이언트가 미처 고려하지 못했을 수 있는 포인트 1개)

포트폴리오와 상세 경험은 프로필에서 확인 가능합니다.
감사합니다.
```

## DO

- 프로젝트별 맞춤 작성 — 복붙 금지, 프로젝트 고유 내용 필수
- 경험 중심 어필 — 기술 디테일보다 "무엇을 해서 어떤 결과를 냈는지"
- 개조식/명사형 종결 — "~했습니다" 반복보다 "~수행", "~해결"
- 830자 이상 (위시켓 데이터봇 권장)
- 포트폴리오 2-3개 첨부 (매칭 확률 60% 상승)
- 모든 경험에 "이슈→해결→결과" 패턴 필수 (나열 금지)
- "미팅 시 논의 희망 사항" 포함
- 기간은 확정형 ("2주") 사용
- **진행 프로세스에 중간 공유 포인트 명시** — "주 1회 진행 공유", "중간 데모 N회" 등 (성공 파트너 평가에서 "소통", "적극성"이 반복 언급)
- **클라이언트가 미처 생각하지 못한 포인트 1개 제안** — "논의 사항"에 포함 (성공 파트너 평가: "생각하지 못한 부분까지도 제안")

## DON'T

- 기술 스택 나열 섹션
- 컴포넌트/훅 이름 (PermissionGate, usePermission 등)
- 기술 버전 (AI SDK v5 등)
- 코드량 지표 (커밋 수, PR 수)
- 범위형 기간 ("1~2주", "2~3주")
- 기술 용어 직접 사용 — 비즈니스 언어로 전환:

| 금지 | 대체 |
|------|------|
| hospital_id, tenant_id | 병원별/테넌트별 데이터 격리 |
| RBAC | 역할별 접근 권한 체계 |
| Chart.js, D3 | 시각화 대시보드 |
| API 레이어 추상화 | 데이터 호출 구조 정리 |
| WebSocket | 실시간 데이터 연동 |
| 가상화 (virtualization) | 대량 데이터 화면 최적화 |
| React Query, Zustand, Jotai | (언급하지 않음) |

## 금액/기간 산정

6단계 분석. 단순히 "예산 동일"이 아니라, 클라이언트 예산의 현실성을 먼저 판단하고, 제안 금액이 예상에서 크게 벗어나지 않도록 조율한다.

### Step 1: 스코프 분석

프로젝트 설명에서:
- 기능 모듈 수 (인증, CRUD, 대시보드, 관리자, 결제 등)
- 업무 범위 (개발만 vs 개발+디자인+기획)
- MVP인지 풀 스코프인지
- 디자인 시안 유무 (있으면 공수 절감)

### Step 2: 공수 추정

모듈 × 복잡도로 인일(man-day) 산정. 참고 기준:

| 작업 유형 | 인일 기준 |
|----------|----------|
| 기획/설계 (ERD, 화면설계) | 3~7일 |
| 인증/권한 모듈 | 3~5일 |
| CRUD 1세트 (목록+상세+폼) | 2~3일 |
| 대시보드/차트 | 3~5일 |
| 결제 연동 (PG) | 3~5일 |
| 관리자 페이지 | 5~10일 |
| 모바일 앱 (RN) | 10~20일 |
| QA/수정/배포 | 전체의 15~20% |

### Step 3: 클라이언트 예산 현실성 판단

위시켓 프로젝트에는 두 가지 금액이 있다:
- **예상 금액**: 클라이언트가 설정한 예산
- **예상 기간**: 클라이언트가 설정한 기간

이 둘로 **클라이언트 기대 일당**을 계산한다:

```
클라이언트 기대 일당 = 예상 금액 ÷ 예상 기간
```

이 값을 Step 2의 공수 추정과 비교하여 현실성을 판단:

| 클라이언트 기대 일당 | 판단 | 대응 |
|-------------------|------|------|
| 15만 미만/일 | 비현실적 예산 | 지원 재고. 지원하더라도 스코프 대폭 축소 필요 멘트 필수 |
| 15~25만/일 | 타이트한 예산 | 지원 가능하나 MVP 범위 한정 또는 기능 우선순위 조율 필요 |
| 25~40만/일 | 적정 예산 | 기본 시나리오. 예산 대비 스코프 균형 확인 |
| 40만 이상/일 | 여유 예산 | 기간 단축 또는 품질 강화로 가치 어필 가능 |

추가로, **공수 추정 vs 클라이언트 기간** 비교:
- 추정 공수 > 클라이언트 기간 × 1.3: "타이트" — 스코프 조율 또는 MVP 논의
- 추정 공수 ≈ 클라이언트 기간: "적정"
- 추정 공수 < 클라이언트 기간 × 0.7: "여유" — 기간 단축 제안 가능

### Step 4: 내 제안 금액/기간 결정

**제안 금액**과 **제안 기간**을 결정한다. 핵심 규칙:

**일당 하한선: 15만원/일** — 이보다 낮게 제안하지 않는다. 단, 인위적으로 낮추지도 않는다.

| 상황 | 금액 전략 | 기간 전략 |
|------|----------|----------|
| 클라이언트 일당 적정(25~40만) + feasibility 4.0+ | 예산 동일 또는 기간 10% 단축 | 기간 단축으로 자신감 어필 |
| 클라이언트 일당 적정(25~40만) + feasibility 3.0~3.9 | 예산 동일, 기간 동일 | — |
| 클라이언트 일당 타이트(15~25만) | 예산 동일 유지 (저가 경쟁 금지) | 기간 동일 + "미팅에서 스코프 조율" 멘트 |
| 클라이언트 일당 여유(40만+) | 기간 단축(-10~15%) | 단축된 기간으로 가치 어필 |
| 클라이언트 일당 비현실적(<15만) | 예산 동일 + 명확한 스코프 한정 | MVP만 명시, 나머지 2차 분리 |
| feasibility 2.0~2.9 | 예산 동일, MVP 범위 한정 | — |

### Step 5: 이탈도 체크

제안 금액/기간이 클라이언트 예상에서 **과도하게 벗어나면** 미팅 성사 확률이 낮아진다.

```
금액 이탈도 = |제안 금액 - 예상 금액| ÷ 예상 금액 × 100%
기간 이탈도 = |제안 기간 - 예상 기간| ÷ 예상 기간 × 100%
```

| 이탈도 | 판단 | 조치 |
|--------|------|------|
| 0~10% | 안전 | 그대로 진행 |
| 10~20% | 주의 | 지원서에서 이유를 자연스럽게 설명 (스코프 조율, 기간 단축 근거) |
| 20%+ | 위험 | 재조정 필요. 20% 이상 벗어나면 클라이언트가 검토조차 안 할 가능성 |

예외: 기간을 10~15% 단축하는 것은 긍정적 이탈 (빠른 납기 = 가치). 금액을 20%+ 높이는 것은 부정적 이탈.

### Step 6: 최종 확인 + 벤치마크 비교

**내 제안 일당** = 제안 금액 ÷ 제안 기간

이 값을 벤치마크와 비교:

| 기준 | 일당 범위 | 근거 |
|------|----------|------|
| 시장 시니어 풀스택 | 25~40만/일 | 위시켓 일반 시세 |
| 성공 파트너 A (12건, 5.0) | 25~30만/일 | 150만/6일 ~ 3000만/99일 |
| 성공 파트너 B (31건, 5.0) | 9~23만/일 | 80만/9일 ~ 850만/70일 (볼륨형) |
| 성공 파트너 C (78건, 4.93) | 다양 | 60만 ~ 1000만 (니치 도메인) |
| **일당 하한선** | **15만/일** | 이보다 낮게 제안하지 않음 |
| **신규 파트너 현실 범위** | **15~33만/일** | 실적 0건, 무리하게 높이지 않되 인위적으로 낮추지도 않음 |

핵심 인사이트:
- 성공 파트너도 초기에는 소규모(150만/6일, 25만/일)로 시작
- **위시켓 데이터봇: 미팅 성사 파트너의 금액이 평균보다 오히려 높았다** → 저가 경쟁은 역효과
- **예산보다 싸게 써서 계약까지 간 경우 14%뿐** → 클라이언트는 금액보다 유사 경험 중시
- **첫 수주 전략**: 500만~1500만 소규모에서 5.0 리뷰 확보 → 이후 단가 상향

### 출력 형식 (작성 근거에 포함)

```
### 금액/기간 산정
- 스코프: {모듈 수, 범위, MVP 여부}
- 공수 추정: {인일} (근거: 모듈별 인일 산정)
- 클라이언트 기대 일당: {예상 금액}÷{예상 기간} = {N}만원/일 → {적정/타이트/여유/비현실적}
- 제안: {금액}/{기간} → 제안 일당 {M}만원/일
- 금액 이탈도: {X}% / 기간 이탈도: {Y}% → {안전/주의/위험}
- 벤치마크 비교: 신규 파트너 범위(15~33만) {이내/초과}
- 최종 결정 근거: {1-2줄}
```

## 성공 파트너 패턴 (지원서 내용에 반영)

위시켓 상위 파트너 3명의 클라이언트 평가에서 반복 등장하는 키워드:

| 패턴 | 빈도 | 지원서 반영 위치 |
|------|------|----------------|
| "적극적" / "적극성" | 모든 파트너 | 진행 프로세스: "주 N회 진행 공유" 명시 |
| "문제해결이 빠르다" | 파트너 A | 유사 경험: "이슈→해결" 패턴으로 어필 |
| "사전 미팅 준비성" / "꼼꼼" | 파트너 B | 프로젝트 분석: 이슈를 미리 정리해서 보여주기 |
| "생각하지 못한 부분까지 제안" | 파트너 A | 미팅 논의: 클라이언트가 놓칠 수 있는 포인트 1개 |
| "수정요청에도 끝까지" | 파트너 B | 진행 프로세스: QA/수정 단계 명시 |
| "클라이언트 입장 고려" | 파트너 B | 전체 톤: 기술 과시 아닌 비즈니스 문제 이해 |

이 패턴들은 템플릿에 이미 구조적으로 녹아있다:
- **[프로젝트 분석]** → "꼼꼼함", "준비성"
- **[유사 경험] "이슈→해결"** → "문제해결 속도"
- **[진행 프로세스] 중간 공유** → "적극성", "소통"
- **[미팅 논의] 추가 제안** → "생각하지 못한 부분 제안"
- **QA/수정 단계 포함** → "수정요청에도 끝까지"
