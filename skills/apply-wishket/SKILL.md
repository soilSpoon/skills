# Apply Wishket

위시켓 프로젝트에 대한 맞춤형 지원서를 생성한다. 단건 또는 다건 병렬 처리 가능.

## Input

사용자가 다음 중 하나를 제공:

1. **프로젝트 URL**: `https://www.wishket.com/project/153999/`
2. **프로젝트 ID**: `153999`
3. **ID 목록**: `153999, 154006, 153297` 또는 파일 경로
4. **"전체"**: available-projects.md에서 미지원 프로젝트 전체

---

## Workflow

### Phase 1: 프로젝트 분석

각 프로젝트 ID에 대해:

**1-1. 프로젝트 상세 Fetch**

WebFetch로 `https://www.wishket.com/project/{ID}/` 를 fetch:
- 프로젝트 상세 설명 전문
- 요구 기술 스택
- 업무 범위 (개발/디자인/기획)
- 클라이언트 질문 (있을 경우)
- 예산, 기간, 모집 마감일, 지원자 수

**1-2. Feasibility 스코어링**

5가지 기준으로 1~5점 가중 평균:

| 기준 | 가중치 | 1점 | 3점 | 5점 |
|------|--------|-----|-----|-----|
| 기술 매칭도 | 30% | 핵심 기술 경험 없음 | 유사 기술 경험 | 동일 스택 실무 경험 |
| 도메인 유사성 | 25% | 완전히 새로운 도메인 | 유사 패턴 존재 | 동일 도메인 경험 |
| 규모 대비 기간 | 20% | 기간 내 불가 | 타이트하지만 가능 | 충분한 여유 |
| 경쟁 수준 | 15% | 50명+ | 20~50명 | 20명 미만 |
| 클라이언트 신뢰도 | 10% | 첫 프로젝트 | 1~2건 | 다수, 높은 평가 |

비공개 지원자 수는 3점 처리. 종합 2.0 미만은 스킵 권장하고 사용자에게 확인.

### Phase 2: 경험 매칭

**2-1. 경험 풀 로드**

`cv/master.yaml`을 읽고 사용 가능한 경험을 추출. 각 경험은 **익명화 코드**로 참조:

| 코드 | 출처 | 핵심 역량 | 사용 가능한 "이슈→해결" 스토리 |
|------|------|----------|-------------------------------|
| HR | HR SaaS 4년 | 5단계 권한, 디자인 시스템, 성능 최적화, jQuery→React 전환, RN 앱 | 권한 경계→선언적 분리, 프리징→가상화+캐싱, 레거시→모던 전환 |
| SIM | 시뮬레이션 플랫폼 | WASM, AI 챗봇, PostHog, 124개국 | 진입장벽→AI 자연어, 32비트→64비트, 데이터 부재→PostHog 구축 |
| MES | 제철소 MES | WebSocket, 규칙 엔진, Electron, 실시간 | 유입 초과→배치 렌더링, 디스크 누적→메모리 파이프라인 |
| TOK | 토큰 시각화 서비스 | 서버리스, 크론 수집, Chart.js | API 포맷 변경→스키마 검증+폴백 |
| OSS | 오픈소스 23개 기여 | webpack, Rust, 메인테이너 | HMR 미지원→소스 분석 후 직접 구현 |
| MENU | AI 메뉴 추천 | Gemini, 프롬프트, Cloud Run | 추천 정확도→태깅+프롬프트 개선, VM→Cloud Run 마이그레이션 |

**2-2. 매칭 규칙**

- 프로젝트 요구사항과 경험 풀의 키워드를 대조하여 관련도 순으로 2-3개 선택
- **동일 경험 코드에서 2개 스토리 추출 금지** — HR에서 권한 이야기를 쓰면 같은 HR에서 성능 이야기를 또 쓰지 않음
- **다건 처리 시**: 10건 전체에서 경험 배분이 편중되지 않도록, 가능하면 다양한 조합 사용
- 각 경험은 **반드시 "이슈→해결→결과" 3줄 구조** 사용 (나열 금지)

### Phase 3: 금액/기간 산정

**단순 "예산 동일"이 아닌 프로젝트별 분석:**

```
Step 1: 스코프 분석
  - 프로젝트 설명에서 주요 기능 모듈 수 파악
  - "개발+디자인+기획" vs "개발만" 범위 확인
  - MVP인지 풀 스코프인지 판단

Step 2: 공수 추정
  - 모듈 수 × 복잡도로 대략적 인일(man-day) 산정
  - 내가 이 기술 스택에서 실제로 걸릴 시간 기준

Step 3: 일당 단가 계산
  - 프로젝트 예산 ÷ 프로젝트 기간 = 일당
  - 시장 기준: 시니어 풀스택 25~40만/일
  - 일당 25만 미만이면 "타이트한 예산" 판단
  - 일당 40만 이상이면 "프리미엄 예산" 판단

Step 4: 최종 결정
  - feasibility 4.0+: 예산 동일 또는 기간 단축(-10%)으로 경쟁력 확보
  - feasibility 3.0~3.9: 예산 동일, 기간 동일
  - feasibility 2.0~2.9: MVP 범위 한정 명시, 예산 동일 유지 (저가 경쟁 금지)
  - 일당이 25만 미만이면: 예산 동일 + 스코프 조율 필요 멘트 추가
  - 일당이 40만 이상이면: 기간 단축으로 가치 어필
```

**핵심 원칙:**
- 저가 경쟁 금지 (위시켓 데이터: 싸게 써서 계약까지 간 경우 14%뿐)
- 기간은 확정형 ("2주") 사용, 범위형 ("1~2주") 금지
- 첫 수주는 확실한 프로젝트에서 5.0 리뷰 확보가 목표

### Phase 4: 지원서 작성

**4단 구조 템플릿:**

```
안녕하세요. React/TypeScript 7년차 풀스택 개발자입니다.

[프로젝트 분석]
본 프로젝트의 핵심은 다음과 같이 파악됩니다:
1. (핵심 1)
2. (핵심 2)
3. (핵심 3)

예상되는 주요 이슈:
- (이슈 1): (왜 이슈인지 한 줄 설명)
- (이슈 2): (왜 이슈인지 한 줄 설명)

[유사 프로젝트 경험]
1) (경험 제목 — 익명화)
- (프로젝트 배경 한 줄)
- 실제로 겪은 이슈: (XX 문제) → (YY 방식으로 해결)
- 결과: (성과/효과)

2) (경험 제목 — 익명화)
- (프로젝트 배경 한 줄)
- 실제로 겪은 이슈: (XX 문제) → (YY 방식으로 해결)
- 결과: (성과/효과)

[진행 프로세스]
1단계. (단계명) — (확정 기간)
- (구체적 내용)
- (산출물/공유 항목)

2단계. (단계명) — (확정 기간)
- (구체적 내용)
- (중간 데모/공유 포함)

3단계. (단계명) — (확정 기간)
- (구체적 내용)

[미팅 시 논의 희망 사항]
- (논의 사항 1)
- (논의 사항 2)

포트폴리오와 상세 경험은 프로필에서 확인 가능합니다.
감사합니다.
```

**작성 규칙:**

DO:
- 프로젝트별 맞춤 작성 (복붙 금지, 프로젝트 고유 내용 필수)
- 경험 중심 어필 (기술 디테일보다 "무엇을 해서 어떤 결과")
- 개조식/명사형 종결
- 830자 이상
- 포트폴리오 2-3개 첨부
- 모든 경험에 "이슈→해결→결과" 패턴
- "미팅 시 논의 희망 사항" 포함

DON'T:
- 기술 스택 나열 섹션
- 컴포넌트/훅 이름 (PermissionGate, usePermission 등)
- 기술 버전 (AI SDK v5 등)
- 코드량 지표 (커밋 수, PR 수)
- 기술 용어 직접 사용 — 비즈니스 언어로 전환:

| 금지 | 대체 |
|------|------|
| hospital_id, tenant_id | 병원별/테넌트별 데이터 격리 |
| RBAC | 역할별 접근 권한 체계 |
| Chart.js, D3 | 시각화 대시보드 |
| API 레이어 추상화 | 데이터 호출 구조 정리 |
| WebSocket | 실시간 데이터 연동 |
| 가상화 (virtualization) | 대량 데이터 화면 최적화 |
| React Query, Zustand, Jotai | (언급하지 않음) |

**익명화 필수:**

| 금지 (실명) | 사용 (익명) |
|------------|-----------|
| 출근노트, gwnote | HR SaaS, 근태/급여 관리 SaaS |
| 에브리드론, everydrone | 시뮬레이션 플랫폼, 드론/CFD SaaS |
| GSS, 포스코 | 제철소 MES 시스템 |
| 슈퍼워크 바닥가 | 토큰 가격 시각화 서비스 |

### Phase 5: 포트폴리오 선택

위시켓에 등록된 포트폴리오 5개에서 2-3개 선택:

| ID | 포트폴리오명 | 매칭 유형 |
|----|------------|----------|
| 277406 | 근태 및 급여 등 HR 관리를 위한 웹 플랫폼 | SaaS, 권한, 관리 시스템, PHP/Laravel |
| 287046 | 클라우드 기반 드론 설계 및 시뮬레이션 플랫폼 | AI, 챗봇, SaaS, Next.js, WASM |
| 291990 | 제철소 실시간 품질 분석 데스크탑 시스템 | 데이터, 대시보드, 실시간, Electron |
| 291992 | 블록체인 토큰 가격 추이 시각화 서비스 | 데이터 시각화, 서버리스, 사이드프로젝트 |
| 287048 | AI를 활용한 식사 메뉴 추천 서비스 | AI, Gen AI, 프롬프트 |

매핑 로직:
- 프로젝트 요구 기술/도메인 → 포트폴리오 매칭 유형 비교
- 항상 2-3개 선택 (1개는 매칭률 60% 하락)
- 가장 관련 높은 것이 첫 번째

### Phase 6: 검증 (자동)

모든 지원서 생성 후 자동 검증:

**체크리스트 (8항목 + 4추가):**

| # | 항목 | 자동 검증 |
|---|------|----------|
| 1 | 프로젝트 핵심 2-3가지 | grep "[프로젝트 분석]" 후 번호 리스트 카운트 |
| 2 | 예상 이슈 2-3개 | grep "이슈:" 카운트 |
| 3 | 모든 경험 "이슈→해결→결과" | grep "겪은 이슈" 카운트 = 경험 수 |
| 4 | 단계별 프로세스 + 확정 기간 | grep "단계" 카운트, 범위형("~") 없는지 확인 |
| 5 | 830자 이상 | 본문 글자 수 |
| 6 | 포트폴리오 2-3개 | portfolio ID 카운트 |
| 7 | 기술 디테일 제거 | 금지어 grep: PermissionGate, usePermission, SDK v5, hospital_id, RBAC |
| 8 | "미팅 시 논의" 포함 | grep "미팅 시 논의" |
| 9 | 익명화 준수 | 금지어 grep: 출근노트, 에브리드론, GSS, 슈퍼워크, gwnote, everydrone, 포스코 |
| 10 | 기간 확정형 | 범위형 ("1~2주", "2~3주") 없는지 확인 |
| 11 | 경험 소스 편중 없음 | 동일 코드에서 2개 스토리 사용 여부 |
| 12 | 금액/기간 근거 | 일당 단가가 시장 범위(25~40만) 내인지 확인 |

**불합격 시:** 해당 지원서만 수정하여 재검증. 전체를 다시 돌리지 않음.

### Phase 7: 사용자 리뷰 요청

검증 통과한 지원서를 요약 테이블로 제시:

```
| # | 프로젝트 | 금액 | 기간 | 일당 | 포트폴리오 | 검증 |
|---|---------|------|------|------|-----------|------|
| 1 | ... | ... | ... | ... | ... | 8/8 |
```

사용자가 승인하면 위시켓에 제출하거나, 수정 요청 시 해당 건만 재생성.

## 다건 병렬 처리

ID가 여러 개일 때:
1. Phase 1 (분석) 은 병렬 WebFetch 가능
2. Phase 2-5 (매칭~작성) 는 **프로젝트당 1개 에이전트**로 병렬 디스패치
3. Phase 6 (검증) 은 **모든 에이전트 완료 후** 단일 검증 에이전트가 전체를 한번에 검사
4. Phase 7 (리뷰) 는 메인 세션에서 사용자와 대화

에이전트 디스패치 시:
- 모든 에이전트를 `run_in_background: true`로 실행
- 각 에이전트에 공유 리소스 경로와 프로젝트 정보를 전달
- 결과물은 `benchmark/wishket/proposals/{ID}.md`에 저장
