# Apply Wishket

위시켓 프로젝트에 대한 맞춤형 지원서를 생성한다.

프로젝트 상세를 분석하고, 사용자의 경험 데이터(master.yaml)에서 가장 관련 높은 경험을 매칭하여, 클라이언트가 "이 사람은 프로젝트를 이해했구나"라고 느끼는 지원서를 작성한다. 생성 후 fact-check로 수치/경험의 정확성을 검증한다.

## Input

사용자가 다음 중 하나를 제공:

1. **프로젝트 URL**: `https://www.wishket.com/project/153999/`
2. **프로젝트 ID**: `153999`
3. **ID 목록**: `153999, 154006, 153297` 또는 파일 경로
4. **"전체"**: 프로젝트 목록 파일에서 미지원 건 전체

URL에서 ID 추출: `/project/(\d+)/` 패턴 매칭.

## Bundled Resources

이 스킬과 함께 번들된 파일들 (`<skill-dir>/` = 이 SKILL.md가 있는 디렉토리):

| 파일 | 용도 | 언제 읽는가 |
|------|------|-----------|
| `<skill-dir>/docs/experience-pool.md` | 경험 코드 테이블, 매칭 규칙, 익명화 규칙, 포트폴리오 매핑 | Phase 2 시작 시 |
| `<skill-dir>/docs/proposal-rules.md` | 지원서 템플릿, DO/DON'T, 금액 산정 로직 | Phase 3-4 시작 시 |
| `<skill-dir>/agents/proposal-writer.md` | 생성 에이전트 프롬프트 (다건 병렬 시) | Wave 1 디스패치 시 |
| `<skill-dir>/agents/verifier.md` | 검증 에이전트 프롬프트 | Wave 2 디스패치 시 |

프로젝트 루트의 파일:
| 파일 | 용도 |
|------|------|
| `cv/master.yaml` | 경험 원천 데이터 (수치, 기간, 기술 스택) |

## Workflow

### Phase 1: 프로젝트 분석

**1-1. 프로젝트 상세 Fetch**

WebFetch로 `https://www.wishket.com/project/{ID}/` 접근하여 추출:
- 프로젝트 상세 설명 전문
- 요구 기술 스택
- 업무 범위 (개발/디자인/기획)
- 클라이언트 질문 (있을 경우)
- 예산, 기간, 모집 마감일, 지원자 수

**1-2. Feasibility 스코어링**

5가지 기준 가중 평균으로 지원 적합도를 판단한다. 이 점수는 금액 전략과 지원 우선순위에 영향을 준다.

| 기준 | 가중치 | 1점 | 3점 | 5점 |
|------|--------|-----|-----|-----|
| 기술 매칭도 | 30% | 핵심 기술 경험 없음 | 유사 기술 경험 | 동일 스택 실무 경험 |
| 도메인 유사성 | 25% | 완전히 새로운 도메인 | 유사 패턴 존재 | 동일 도메인 경험 |
| 규모 대비 기간 | 20% | 기간 내 불가 | 타이트하지만 가능 | 충분한 여유 |
| 경쟁 수준 | 15% | 50명+ | 20~50명 | 20명 미만 |
| 클라이언트 신뢰도 | 10% | 첫 프로젝트 | 1~2건 | 다수, 높은 평가 |

비공개 지원자 수는 3점 처리. **종합 2.0 미만은 스킵 권장** — 사용자에게 확인 후 진행.

### Phase 2: 경험 매칭

`<skill-dir>/docs/experience-pool.md`를 읽고 경험 코드 테이블을 로드한다.
`cv/master.yaml`을 읽고 실제 수치/기간/스토리를 확인한다.

프로젝트 요구사항과 경험 풀을 대조하여 2-3개를 선택한다. 매칭 규칙과 익명화 규칙은 experience-pool.md에 정의되어 있다.

경험을 선택할 때 중요한 것은 기술 키워드 일치가 아니라, 클라이언트가 "이 사람이 우리 프로젝트의 어려운 점을 이미 경험해봤구나"라고 느낄 수 있는 스토리를 고르는 것이다. 프로젝트의 예상 이슈와 직접 연결되는 경험이 최우선.

### Phase 3: 금액/기간 산정

`<skill-dir>/docs/proposal-rules.md`의 금액 산정 섹션을 읽고 4단계 분석을 수행한다.

핵심: "예산 동일"로 끝내지 않고, 스코프 → 공수 → 일당 → 시장 비교 → 최종 결정의 근거 체인을 남긴다. 클라이언트 예산이 비현실적일 수 있으므로, 일당 단가가 시장 범위(25~40만)를 벗어나면 지원서에서 스코프 조율이나 우선순위 논의를 자연스럽게 언급한다.

### Phase 4: 지원서 작성

`<skill-dir>/docs/proposal-rules.md`의 템플릿과 DO/DON'T를 따라 4단 구조 지원서를 작성한다.

지원서의 목적은 "이 개발자가 우리 프로젝트를 이해하고 있고, 비슷한 문제를 풀어본 경험이 있으며, 체계적으로 진행할 수 있다"는 인상을 주는 것이다. 기술적 깊이를 과시하는 것이 아니라, 클라이언트의 비즈니스 문제를 이해하고 있다는 것을 보여준다.

### Phase 5: 포트폴리오 선택

`<skill-dir>/docs/experience-pool.md`의 포트폴리오 매핑 테이블에서 2-3개를 선택한다. 항상 2개 이상 — 1개만 첨부하면 매칭률이 60% 하락한다(위시켓 데이터봇 통계).

### Phase 6: 검증

생성된 지원서에 대해 2단계 검증을 수행한다.

#### 6-1. 구조 검사

아래 12항목을 확인한다:

| # | 항목 |
|---|------|
| 1 | 프로젝트 핵심 2-3가지 파악 |
| 2 | 예상 이슈 2-3개 |
| 3 | 모든 경험 "이슈→해결→결과" 패턴 (나열형 0건) |
| 4 | 단계별 프로세스 + 확정형 기간 (범위형 금지) |
| 5 | 830자 이상 |
| 6 | 포트폴리오 2-3개 |
| 7 | 기술 디테일 금지어 없음 |
| 8 | "미팅 시 논의 희망 사항" 포함 |
| 9 | 익명화 금지어 없음 |
| 10 | 범위형 기간 ("1~2주") 없음 |
| 11 | 동일 경험 코드에서 2개 스토리 없음 |
| 12 | 금액 근거 (일당 단가 + 시장 비교) 기재 |

#### 6-2. Fact-Check

`<skill-dir>/agents/verifier.md`를 읽고 검증 에이전트를 실행한다.
지원서의 수치/경험이 master.yaml과 일치하는지 대조한다.

- **PASS**: 정확히 일치
- **FAIL**: 수치 불일치 (과장, 축소, 날조) → 해당 부분 수정 후 재검증
- **WARN**: 정성적 주장 (검증 불가) → 사용자에게 보고

### Phase 7: 사용자 리뷰

검증 통과한 지원서를 요약 테이블로 제시:

```
| # | 프로젝트 | 금액 | 기간 | 일당 | 포트폴리오 | 구조 | Fact |
|---|---------|------|------|------|-----------|------|------|
| 1 | ... | ...만 | ...일 | ...만/일 | 2-3개 | 12/12 | PASS |
```

각 지원서별 1줄 요약 (핵심 매칭 근거). WARN 항목이 있으면 별도 표시.
사용자 승인 시 위시켓에 제출, 수정 요청 시 해당 건만 재생성.

## 에이전트 아키텍처 (다건 처리)

프로젝트가 2건 이상이면 병렬 에이전트를 활용한다.

```
[Wave 1] 생성 — N개 병렬 (background)
  각 에이전트: agents/proposal-writer.md 프롬프트로 Phase 1~5 독립 수행
  → proposals/{ID}.md 저장

  (전체 완료 대기)

[Wave 2] 검증 — 에이전트 1개 (background)
  agents/verifier.md 프롬프트로 전체 proposals/ + master.yaml 대조
  → 수치 검증 + 크로스 일관성 검사

  (완료 대기)

[Wave 2.5] 구조 검사 — 메인 세션에서 직접 수행

[Wave 3] 수정 — FAIL 건만 (있으면 병렬)
  FAIL 사유 + 파일 경로 전달 → 해당 부분만 수정

Phase 7: 사용자 리뷰 — 메인 세션
```

**디스패치 규칙:**
- Wave 1: 단일 메시지에 N개 Agent 호출, `run_in_background: true`
- Wave 2: Wave 1 전체 완료 후 실행
- 각 에이전트에 전달: 프로젝트 ID, 공유 리소스 경로, `<skill-dir>` 경로

**단건이면** 에이전트 없이 메인 세션에서 Phase 1~7을 직접 수행.
