# 좋은 에러 메시지 6원칙 — Navigating

출처: 토스 "좋은 에러 메시지를 만드는 6가지 원칙"(toss.tech/article/21021).

## 최고 가치: Navigating
**"사용자가 다음 단계로 무엇을 해야 할지 안내하고 있는가."** 에러 메시지는 실패를
*통보*하는 곳이 아니라 사용자를 *해결로 안내*하는 곳이다. 좋은 에러는 세 가지를 담는다:

1. **상황** — 지금 어떤 상황인지
2. **이유** — 왜 발생했는지
3. **해결책** — 어떻게 하면 되는지

세 가지가 다 필요하진 않지만, **해결책(다음 행동)은 거의 항상 있어야 한다.**

## 6원칙 (우선순위 순 — 위가 먼저)

### 1. 최고의 에러는 발생하지 않는 것
문구를 쓰기 전에 **이 에러가 꼭 필요한가**를 먼저 묻는다. UX로 에러 상황 자체를 없애는 게
1순위. 에러 메시지는 차선책이다.
- 연락처 송금에서 "자신에게 송금 불가" 에러를 띄우기보다, 애초에 목록에서 자기 자신을 뺀다.
- 폼 검증: 입력이 불가능하게 막거나(숫자만 입력) 실시간 안내로 제출 전 에러를 없앤다.

### 2. 적절한 컴포넌트 쓰기
에러의 **심각도·설명량**에 맞는 그릇을 고른다.
- **토스트**(3–5초 자동 소멸): 짧고 가벼운 안내. 복잡한 설명·중요한 차단에는 부적합.
- **다이얼로그**: 상황·이유·해결책을 충분히 설명해야 할 때, 사용자의 확인/행동이 필요할 때.
- **인라인**(필드 아래): 폼 검증처럼 *어디가* 문제인지 위치가 중요할 때.
- ❌ 토스트로 "심사가 시작돼서 대출을 취소할 수 없어요" (사라져서 못 읽고, 해결책도 없음)
- ✅ 다이얼로그에 상황+이유+해결책

### 3. 스스로 해결할 방법 알려주기
에러의 가장 중요한 역할. 구체적인 해결책을 준다.
- ❌ "생체 인증 실패"
- ✅ "생체 인증에 실패했어요. 앱을 다시 켜거나 다시 설치하면 해결돼요."

### 4. 사용자 입장의 언어로 쓰기
공급자·개발자 관점의 기술 용어·내부 코드를 버린다. (→ `ux-rules.md` #7과 같은 뿌리.)
- ❌ "예외 처리된 신분증입니다" → ✅ "기간이 만료된 신분증이에요"
- ❌ "Error 4012 / Null geometry" → ✅ "형상을 불러오지 못했어요. 파일을 다시 올려 주세요."

### 5. 쉽게 해결하게 도와주기
해결책을 *말*로만 주지 말고, 그 행동을 **한 번에 할 수단**(버튼·링크)을 붙인다.
- ❌ "고객센터로 문의해 주세요" (앱을 떠나 번호를 찾아야 함)
- ✅ 본문 + **[고객센터 문의]** 버튼 (탭 한 번)
- ❌ "다시 로그인해 주세요" → ✅ + **[로그인하기]** 버튼

### 6. 부정적인 감정 최소화하기
사용자는 기대한 화면을 못 봐서 이미 실망했다. 부정형을 긍정형으로, 강요를 권유로.
- ❌ "본인 인증을 해야 해요" → ✅ "본인 인증을 하면 바로 쓸 수 있어요"
- 책임을 사용자 탓으로 돌리지 않는다: "잘못 입력하셨어요" → "다시 한 번 확인해 주세요"

## 에러 문구 골격

```
[상황 — 무엇이 일어났는가, 사용자 언어로]
[이유 — 필요하면, 짧게]
[해결책 — 다음 행동] + [그 행동을 실행하는 버튼]
```

예) **형상을 불러오지 못했어요**
지원하지 않는 파일 형식이에요. STEP·IGES·STL 파일을 올려 주세요.
**[파일 다시 올리기]**

## 빈 상태(empty state)·성공도 같은 뿌리
- **빈 상태**: "데이터 없음"(통보) → "아직 분석 결과가 없어요. 시뮬레이션을 실행하면 여기에
  나타나요." + 행동 버튼. (Navigating은 에러만의 것이 아니다.)
- **성공/완료**: 사실 통보를 넘어 감정을 읽는다(프린시플 8) — "완료"보다 노력 인정·다음 안내.
