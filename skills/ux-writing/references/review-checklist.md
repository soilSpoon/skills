# 리뷰 체크리스트 — 문구·i18n 진단

문구를 리뷰할 때 이 순서로 *진단*한 뒤, 해당 reference의 before/after 패턴으로 **하나의
개선안**(여러 안 나열 금지, 이유 한 줄 첨부)을 낸다. 짧은 문구는 1~2단계로 충분하다.

## 0. 유형 분류 (먼저)
이 문구는 무엇인가? → 해당 reference를 편다.
- 버튼/CTA · 톤 · ko 문장 → `ux-rules.md`
- 에러/검증/실패/빈 상태 → `error-messages.md`
- 마케팅/배너/푸시 → `marketing-copy.md`
- 전반 보이스앤톤·강요/과장 의심 → `principles.md`

## 1. 사용자 본위 (가장 큰 그물)
- [ ] **다음 행동**이 담겼나? (에러·빈 상태·검증에서 상태만 통보하고 끝나지 않는가)
- [ ] **내부/개발 용어**가 새지 않았나? (패키지명·라이브러리·시스템 상태·jargon: engine·
      OCCT·FaceID·Bonded·solid·compile·null·exception) → `ux-rules.md` #7
- [ ] 공급자 머릿속 순서가 아니라 **처음 보는 사용자 시점**인가?
- [ ] **강요·불안·과장**으로 행동을 짜내지 않는가? (존중)

## 2. 5 코어밸류
- [ ] **명확** — 두 가지로 해석될 여지가 없나?
- [ ] **간결** — 빼도 뜻이 통하는 단어가 있나? (Weed cutting)
- [ ] **친근** — 소리 내어 읽으면 입에 붙나? 전문용어·번역체는 없나?
- [ ] **존중** — 선택권을 주나? "~해야"보다 "~할 수 있어요"인가?
- [ ] **공감** — 이 순간 사용자의 감정(실망·불안·성취)을 헤아렸나?

## 3. 한국어 기계 규칙 (ko 문구)
- [ ] 해요체로 통일됐나? ("~합니다"·반말 혼입 없나)
- [ ] 능동형인가? ("되어요"→"돼요", 불필요한 수동 없나)
- [ ] 긍정형인가? (없어요/안 돼요 → 할 수 있어요)
- [ ] 과잉 존대 없나? (사물 존대·"~시겠어요?" 남발)
- [ ] 명사 나열(한자어 스택) 풀었나?
- [ ] 버튼: 좌측이 "취소" 아닌 "닫기"인가? 동사로 결과를 예고하나?
- [ ] **예외 표** 위반 아닌가? (서비스 종료 수동형 등은 정당) → `ux-rules.md`

## 4. i18n 건전성 (메시지 파일을 만질 때)
- [ ] **하드코딩 우회**: 사용자 노출 문구가 코드 리터럴이 아니라 `t('...')` 키인가?
- [ ] **en↔ko 쌍**: 한 로케일만 고치지 않았나? 누락 키로 폴백 영어가 뜨지 않나?
- [ ] **한영 혼용**: "Solid를 먼저…"·"zone 설정" 같은 미번역 잔재 없나?
- [ ] **용어 일관성**: 한 개념이 키마다 다르게 번역되지 않았나(유입 방향 vs 유동 방향)?
      → 용어집을 만들고 맞춘다.
- [ ] **톤 일관성**: 비슷한 요소가 ~하세요/~합니다/~중/~했어요로 흩어지지 않았나?

## 진단 출력 형식
```
[문구] "현재 문구"
[유형] 버튼 / 에러 / 마케팅 / …
[문제] 위반한 원칙 1~2개 (예: Navigating 없음 + 내부용어 노출)
[개선] "고친 문구"            ← 단일 안
[이유] 한 줄                 ← 왜 더 나은지 (사용자가 무엇을 더 빨리/정확히/편안히 하나)
```

> 큰 표(수십 개 문구)를 한꺼번에 리뷰할 땐: 먼저 **공통 문제를 그룹핑**(예: "에러 12개가
> 다 해결책 없음", "버튼 8개가 영문 하드코딩")하고, 패턴별로 일괄 개선안을 낸다. 개별
> 나열보다 *규칙*을 세우는 게 i18n 전반 교정엔 더 빠르다.
